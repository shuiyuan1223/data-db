## 2. Prompt 工程优化

### 2.1 优化项一：三维度 Query 类型体系

**问题**: 单一类型 query 无法覆盖不同个性化层次。

**优化方案**: 在每个 Domain 的 SYSTEM_PROMPT 中定义三种 Query 类型：

| 类型 | 定义 | data_fields | 用户意图示例 |
|------|------|-------------|------------|
| Type 1 通用 | 不涉及个人数据的知识查询 | 无 | "高血压的诊断标准是什么" |
| Type 2 显性 | 直接请求查看个人数据 | 1-3 个字段 | "帮我看看我上周的平均心率" |
| Type 3 隐性 | 隐含需要个人数据做分析 | 2-4 个字段 | "最近运动后总觉得累" |

每个类别严格生成 10+10+10 = 30 条。

**效果**: 49,200 条类型分布 1:1:1 严格均衡，去重率仅 0.62%

### 2.2 优化项二：运动 Sparse 组合策略

**问题**: 仅单项运动无法覆盖真实用户多运动场景。

**优化方案**: 非完全排列组合的 Sparse 策略——基于运动科学的类间合理组合：
- 单项: 160 个独立运动项目（15.3%）
- 跨项: 883 个类间组合（84.7%），如有氧+力量（跑步+力量训练）、球类跨项（篮球+足球）

**效果**: 1,043 标签 × 30 条 = 31,290 条运动 query

### 2.3 优化项三：健康特征组合数学

**问题**: 单指标 query 无法覆盖多指标关联的复杂健康查询。

**优化方案**: 13 个基础健康特征（体温/睡眠/血压/心率/血氧/血糖/体重/压力/步数/运动/情绪/月经/综合报告）的数学组合：
- C(13,1) = 13 单特征 → 390 条
- C(13,2) = 78 双特征 → 2,340 条
- C(13,3) = 286 三特征 → 8,580 条

**效果**: 377 标签 × 30 = 11,310 条，双/三特征组合占比 96.5%

### 2.4 优化项四：数据字段目录硬约束

**问题**: LLM "字段幻觉"——生成不在预定义目录中的虚假健康数据字段。

**优化方案**: 在 Prompt 中嵌入完整的数据字段目录，硬约束 LLM 只能从中选取：
- general_medical: 101 个字段（8 大类），覆盖基础体征/体成分/每日汇总/心率压力/睡眠/健康档案/营养/高频时序
- health: 52 个字段，覆盖心脏/睡眠/血压/情绪/体温/营养/活动/女性生殖健康
- sports: SPORTS_SPECIFIC_INDICATORS 字典，30+ 运动项目的专属指标（跑步：配速/步频/触地时间/垂直振幅；骑行：FTP/功率/踏频；游泳：SWOLF/划水频率；高尔夫：挥杆速度/差点...）

**效果**: Checker "格式违规" 缺陷率显著下降

### 2.5 优化项五：5 个 Domain 独立 Prompt 体系

**问题**: 不同数据维度需要完全差异化的生成策略和 Prompt 结构。

**优化方案**: 每个 Domain 的 prompt 文件实现统一接口（`build_persona_block()` + `build_messages()`），但内部逻辑完全独立：

| Domain | Prompt 文件 | 独特 Prompt 设计 |
|--------|-----------|----------------|
| general_medical | `general_medical.py` | DATA_FIELD_CATALOG(101字段) + 术语适配约束（根据 persona 医学知识水平调整） |
| health | `health.py` | HEALTH_DATA_MARKDOWN 健康指标词典 + 下划线分隔多域主题（如"心脏健康_睡眠健康"） |
| sports | `sports.py` | REQUIRES_PERSONA=False，`_build_system_prompt()` 动态注入运动专属指标，LLM 自行生成 15 个 persona |
| sports_health | `sports_health.py` | 跨域 persona 渲染（preferred_sports + sport_mastery + sport_goal + core_health_concerns） |
| red_team | `red_team.py` | 4 套独立模板：injection(攻击难度分级) / induction(6种诱导技术：逻辑劫持/设定覆写/情感投射/语言伪装/角色植入/道德困境) / inquiry(6维功能探测) / clarification(歧义类型+澄清要点) |

**效果**: 每个 Domain 各约 500 条，可独立持续生成，互不影响

### 2.6 优化项六：Checker 审查驱动的 Prompt 迭代闭环

**方法**: Checker 的 `_extract_domain_info(domain)` 自动从 prompt 模块提取 SYSTEM_PROMPT 和 query_type 定义，`_build_review_prompt()` 将其注入审查 prompt。审查结果直接映射到 Prompt 修改方向：

| Checker 发现 | 占比 | Prompt 优化方向 | 具体修改位置 |
|-------------|------|---------------|------------|
| 语言不自然 | 47.4% | 约束术语层次与 persona 背景匹配 | SYSTEM_PROMPT 增加术语适配规则 |
| 类型不匹配 | 31.6% | 强化三种 Query 类型的边界定义 | SYSTEM_PROMPT 中 Type 定义增加判别示例 |
| 逻辑矛盾 | 15.8% | 加强 query 与指定主题的锚定 | USER_TEMPLATE 增加主题关联约束 |
| 重复/相似 | 5.3% | 增加多样性约束 | SYSTEM_PROMPT 增加多样性要求 |

**闭环**: 审查 → 定位问题 → 修改 Prompt → `generator.py` 重新生成 → Checker 再审查 → 正常率持续提升

## 3. 谷歌数据多维度场景适配优化

### 3.1 优化背景

谷歌数据集中的 query 深度嵌入 Google/Fitbit 生态，**不能简单做关键词替换**——需要理解竞品生态的产品能力、指标体系和评估模型，找到华为生态中语义等价的表述。

### 3.2 五维度审视角度

| 维度 | 审视角度 | 分析与适配 |
|------|---------|-----------|
| **品牌生态绑定** | 识别与 Google/Fitbit 产品直接绑定的表述 | Fitbit → Huawei Watch。确保改写后的 query 在华为产品能力范围内逻辑自洽 |
| **AI 功能命名** | 区分竞品 AI 功能名称与华为对应功能的差异 | Fitbit LLM → Huawei Assistant。理解两者功能边界差异，避免不支持的交互场景 |
| **专有指标体系** | 深入分析竞品独有指标的**定义与计算逻辑** | Active Zone Minutes 是 Fitbit 基于心率区间计算的独有指标（低/中/高心率区间按 1x/2x 加权累计）→ 映射为 Intensity Activity Minutes（中高强度活动时间），华为/通用的语义等价概念 |
| **健康评估模型** | 理解竞品评估模型含义，找到**等价概念** | Readiness Score 是 Fitbit 综合 HRV/睡眠质量/近期活动量的恢复评估分数 → Recovery Status（恢复状态），华为使用"恢复时间""跑力指数"等相近概念 |
| **隐含场景语境** | LLM 辅助识别肉眼难以发现的隐含竞品关联 | "Fitbit 追踪睡眠数据的具体算法" → 不是替换 Fitbit 为 Huawei Watch，而是重构为华为 Watch 睡眠监测场景 |

### 3.3 适配原则

1. **语义优先**: 理解 query 背后的用户意图，找到华为生态中语义等价的表述
2. **指标溯源**: 竞品专有指标需追溯其定义和计算逻辑，再选最接近的华为/通用概念
3. **场景自洽**: 改写后的 query 在华为 Watch + Huawei Health 场景下逻辑成立
4. **LLM 辅助发现**: 批量扫描识别非直接命名但与 Google 生态相关的隐含场景

### 3.4 效果

- 4,170 条 → 7,424 条（含扩展）
- 5 个维度系统性适配，非机械文本替换

## 4. UMAP+HDBSCAN 聚类参数调优优化

### 4.1 优化背景

本人在团队提供的聚类框架基础上，负责参数的系统性调优。UMAP（Uniform Manifold Approximation and Projection）负责将高维 embedding 降至低维空间，HDBSCAN（Hierarchical Density-Based Spatial Clustering of Applications with Noise）在低维空间中进行密度聚类。目标是在**噪声率、簇数量、簇粒度**三者之间找到最优平衡。

### 4.2 调优全过程（9 轮实验）

#### 阶段一：初始基线 → 噪声过多

```bash
uv run hqb cluster --umap-n-neighbors 15 --umap-n-components 64 \
  --umap-min-dist 0.0 --hdbscan-min-cluster-size 5 \
  --hdbscan-min-samples 3 --hdbscan-epsilon 0.03 --hdbscan-method eom --parallel 7
# → 2,054 clusters, 17,859 noise (~30%)
```

**分析**: epsilon=0.03 对密度要求过于严格（HDBSCAN 中 epsilon 控制簇合并的距离阈值），大量边缘点被判为噪声。min_samples=3 进一步要求核心点周围至少有 3 个邻居，双重严格条件导致 30% 的样本未被聚类。

#### 阶段二：降噪探索 → 发现关键参数

| 实验 | 关键调整 | 噪声率 | 发现 |
|------|---------|--------|------|
| 方案A | eps↑0.06, min_samples↓1 | ~16% | **min_samples=1 是降噪关键**（核心点判定放宽到只需自身即可） |
| 方案B | eps↑0.08, min_samples↑2 | ~20% | 保守的 min_samples 参数反而噪声升高 |

#### 阶段三：高维方案收敛 → v1

```bash
uv run hqb cluster --umap-n-neighbors 15 --umap-n-components 64 \
  --umap-min-dist 0.0 --hdbscan-min-cluster-size 3 \
  --hdbscan-min-samples 1 --hdbscan-epsilon 0.08 --hdbscan-method eom --parallel 7
# → 1,899 clusters, 4,843 noise (~8%) ← v1 推荐
```

**分析**: `min_dist=0.0`（UMAP 允许嵌入点完全重合）+ `min_samples=1` + `epsilon=0.08` 的组合在高维(64)空间达到最佳平衡。

#### 阶段四：低维度探索

将 UMAP n_components 从 64 降至 12，探索信息压缩后的聚类效果：

| 实验 | n_comp | min_samp | eps | 簇数 | 噪声率 | 发现 |
|------|--------|----------|-----|------|--------|------|
| — | 12 | 6 | 0.18 | 365 | ~17% | min_samples 过大导致簇过少 |
| **v2** | 12 | 1 | 0.18 | 883 | ~5% | 噪声最低但出现超大簇需细分 |
| — | 12 | 1 | 0.08 | 3,516 | ~17% | eps 过小又导致噪声反弹 |
| **v3** | 12 (min_dist=0.0) | 1 | 0.08 | 1,823 | ~8% | 低维均衡方案 |

### 4.3 参数影响规律

| 参数 | 降低 → | 增加 → | 原理 |
|------|--------|--------|------|
| epsilon | 更多噪声 | 更少噪声 | epsilon 是簇合并的距离阈值，越大越容易合并 |
| min_samples | 更少噪声 | 更多噪声 | 核心点需要的邻居数，越大越难成为核心点 |
| min_cluster_size | 更少噪声 | 更多噪声 | 允许的最小簇大小，越大则小簇变噪声 |
| min_dist (UMAP) | 点聚集，边界模糊 | 点分散，识别清晰 | 控制嵌入空间中点的最小距离 |
| n_components (UMAP) | 更少簇(粗粒度) | 更多簇(细粒度) | 降维目标维度，越高保留信息越多 |

### 4.4 最终方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **v1** (64维) | 1,899 簇粒度适中，~8% 噪声 | 簇数偏多 | 细粒度主题分析 |
| v2 (12维) | 最低噪声 5%，883 簇 | 超大簇需二次聚类 | 粗粒度+递归聚类 |
| **v3** (12维) | 1,823 簇均衡，~8% 噪声 | — | 通用方案 |

### 4.5 聚类准确率验证

| 指标 | 值 |
|------|---|
| 总样本数 | 6,734 |
| 总合理数 | 4,523 |
| **全局准确率** | **67.2%** |

### 4.6 调优成果

| 指标 | 优化前 | 优化后 | 改进 |
|------|--------|--------|------|

## 5. 统一审查框架优化

### 5.1 优化背景

面对 49,200+ 条生成数据 + 5 个独立 Domain，传统人工逐条审查完全不可行。

### 5.2 Checker 自适应框架实现

Checker（`checker.py`）的核心优化是自动适配机制：

```
_extract_domain_info(domain)
  → 通过 _DOMAIN_PROMPT_MAP 找到 prompt 模块
  → 提取 SYSTEM_PROMPT + 判断 has_persona(是否 REQUIRES_PERSONA)
  → 返回 domain_info dict

_build_review_prompt(domain_info, queries_batch, domain)
  → 将 SYSTEM_PROMPT 中的生成约束注入审查 prompt
  → LLM reviewer 无需额外说明即可理解原始质量标准

_write_report()
  → 按 Query 类型 / 知识主题(或攻击类型) / 问题类型 三维聚合
  → 输出结构化报告
```

**6 维度缺陷分类**:
- 语言不自然 — 过于书面化/专业化，不符合 persona 表达习惯
- 类型不匹配 — query 实际意图与标注 query_type 不一致
- 逻辑矛盾 — query 与指定主题或 persona 属性矛盾
- 格式违规 — 含禁止符号或格式问题
- 内容重复 — 同 persona×topic 下句式/意图高度相似
- 质量低下 — intent 或 data_fields 与 query 不匹配

### 5.3 审查效果验证

| 维度 | 结果 |
|------|------|
| 总体正常率 | 81.0% (general_medical 抽样 100 条) |
| 最高频问题 | 语言不自然 (47.4%) → 指导术语控制 |
| 次高频问题 | 类型不匹配 (31.6%) → 指导类型边界 |
| 最高风险主题 | 运动损伤预防与康复 (26.7%) → 定向优化 |
| 最稳定类型 | 知识计算 Query (9.1%) → 作为标杆 |

### 5.4 流程价值

- **新 Domain 审查成本**: 从"编写审查规则"降至**零代码**（`_extract_domain_info` 自动适配）
- **审查效率**: 100 条/数分钟 vs 人工数小时
- **优化闭环**: 审查→发现问题→Prompt 修改→重新生成→再审查

## 6. 架构可扩展性优化

### 6.1 Domain 可插拔架构

新增任何数据维度的边际成本趋近于零：

```
新 Domain 接入：
  1. prompts/ 新建 py 文件 → 实现 build_persona_block() + build_messages()
  2. output_db.py → 在 _DOMAIN_TABLE_MAP 注册表 DDL + insert 函数
  3. generator.py → 在 _DOMAIN_PROMPT_MAP 注册 prompt 模块路径
  4. Checker → _extract_domain_info 自动适配，无需修改
──────────────────────────────
全链路自动打通: 生成(generator) → 存储(output_db) → 审查(checker)
```

**已验证**: 5 个 Domain（general_medical / health / sports / sports_health / red_team）均通过此流程接入。

### 6.2 系统工程保障

| 优化项 | 实现方案 | 效果 |
|--------|---------|------|
| 并发生成 | asyncio + Semaphore(batch) + Lock(DB写入) | 49,200 条大批量无竞争 |
| 断点续传 | `is_generated()`/`is_topic_generated()` 查已有数据 | 中断后 <5s 恢复 |
| 指数退避 | `delay = min(2×2^(n-1), 30)` 秒, max_retries=100, timeout=120s | API 限流自动恢复 |
| 失败记录 | `_record_failure()` → failed_tasks 表 | domain/persona/topic/error/attempts/timestamp |
| JSON 模式 | response_format=json_object + markdown fence 自动剥离 | LLM 输出稳定解析 |
| WAL 模式 | SQLite PRAGMA journal_mode=WAL + foreign_keys=ON | 并发读写安全 |

## 7. 优化效果总结

### 7.1 关键指标

| 指标 | 初始/优化前 | 最终/优化后 | 改进 |
|------|-----------|-----------|------|
| 聚类噪声率 | ~30% | ~8% | **-73%** |
| 聚类全局准确率 | — | 67.2% | 基线建立 |
| LLM 去重损失率 | — | 0.62% | 极低 |
| 运动类别覆盖 | 单项 160 | 单项+组合 1,043 | **+552%** |
| 健康类别覆盖 | 单特征 13 | 单+双+三 377 | **+2,800%** |
| 新 Domain 接入成本 | 数百行代码 | 1 prompt + 2 行注册 | **趋近于零** |
| 新 Domain 审查成本 | 编写审查规则 | `_extract_domain_info` 自动适配 | **-100%** |
| Checker 审查正常率 | — | 81% (general_medical) | 首轮即定位 |

### 7.2 优化方法论

1. **Prompt 工程前置**: 三维度类型 + 字段目录硬约束(101/52字段) + 运动指标词典(30+运动) + 组合数学，从生成阶段最大化保证质量
2. **多维度场景适配**: 谷歌数据不做机械替换，从品牌/AI能力/指标定义/评估模型/场景语境 5 个维度理解竞品生态后进行语义等价适配
3. **系统性参数调优**: 9 轮 UMAP+HDBSCAN 实验，建立 epsilon/min_samples/min_dist/n_components 等参数与噪声率/簇数的定量关系
4. **自动化审查闭环**: Checker 通过 `_extract_domain_info` 自适应任意 Domain，6 维度缺陷分类 + 三维聚合报告，直接输出 Prompt 优化方向
5. **可插拔架构设计**: Domain 维度扩展只需 build_persona_block + build_messages + 表注册，边际成本趋近于零
