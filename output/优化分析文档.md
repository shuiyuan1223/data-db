## 1. 优化概述

## 2. Prompt 工程优化

### 2.1 优化项一：三维度 Query 类型体系

**问题**: 单一类型 query 无法覆盖不同个性化层次。

**优化方案**: 每类别生成 10 通用 + 10 显性个性化 + 10 隐性个性化 = 30 条
- 通用: 不涉及个人数据
- 显性个性化: 直接意图涉及个人数据
- 隐性个性化: 隐含意图涉及个人数据

**效果**: 49,200 条类型分布 1:1:1 严格均衡，去重率仅 0.62%

### 2.2 优化项二：运动 Sparse 组合策略

**问题**: 仅单项运动无法覆盖多运动场景。

**优化方案**: 单项(160) + 基于运动科学类间关系的跨项组合(883) = 1,043 标签

**效果**: 31,290 条运动 query，覆盖有氧+力量、球类跨项等真实多运动场景

### 2.3 优化项三：健康特征组合数学

**问题**: 单指标 query 无法覆盖多指标关联的复杂健康查询。

**优化方案**: 13 个基础健康特征 → C(13,1) + C(13,2) + C(13,3) = 377 标签

**效果**: 11,310 条健康 query，双/三特征组合占比 96.5%

### 2.4 优化项四：数据字段目录硬约束

**问题**: LLM"字段幻觉"——生成不在目录中的虚假健康数据字段。

**优化方案**: 在 `general_medical.py` 中定义 100+ 数据字段目录（DATA_FIELD_CATALOG），嵌入 SYSTEM_PROMPT 硬约束。

**效果**: Checker "格式违规" 缺陷率显著下降

### 2.5 优化项五：5 个 Domain 独立 Prompt 体系

**问题**: 不同数据维度需要差异化的生成策略。

**优化方案**: 为每个 Domain 设计**独立的 Prompt 模板文件**，各有专属的生成逻辑：

| Domain | Prompt 文件 | 独特设计 |
|--------|-----------|---------|
| general_medical | `prompts/general_medical.py` | 100+ 数据字段目录硬约束 |
| health | `prompts/health.py` | topic_count 多子域主题 |
| sports | `prompts/sports.py` | LLM 内联 persona（无需外部人设） |
| sports_health | `prompts/sports_health.py` | 跨域 persona 健康+运动融合 |
| red_team | `prompts/red_team.py` | 4 类独立攻击（injection/induction/inquiry/clarification） |

**效果**: 每个 Domain 各约 500 条，可独立持续生成，互不影响

### 2.6 优化项六：Checker 审查驱动的 Prompt 迭代

**方法**: 利用 Checker 抽样审查结果反馈 Prompt 优化方向。

**general_medical 审查结果→优化映射**:

| Checker 发现 | 占比 | Prompt 优化方向 |
|-------------|------|---------------|
| 语言不自然 | 47.4% | 约束术语层次与 persona 背景匹配 |
| 类型不匹配 | 31.6% | 强化三种 Query 类型的边界定义 |
| 逻辑矛盾 | 15.8% | 加强 query 与指定主题的锚定 |
| 重复/相似 | 5.3% | 增加多样性约束 |

**闭环价值**: 审查 → 定位问题 → 修改 Prompt → 重新生成 → 再审查 → 正常率持续提升

## 3. 谷歌数据多维度场景适配优化

### 3.1 优化背景

谷歌数据集中的 query 深度嵌入 Google/Fitbit 生态，**不能简单做关键词替换**，需要从多个维度理解竞品生态语境，再进行语义等价的场景适配。

### 3.2 5 维度审视角度

| 维度 | 审视角度 | 分析与适配 |
|------|---------|-----------|
| **品牌生态绑定** | 识别与 Google/Fitbit 产品直接绑定的表述 | Fitbit → Huawei Watch。确保改写后的 query 在华为产品能力范围内逻辑自洽 |
| **AI 功能命名** | 区分竞品 AI 功能名称与华为对应功能的差异 | Fitbit LLM → Huawei Assistant。理解两者功能边界差异，避免生成不支持的交互场景 |
| **专有指标体系** | 深入分析竞品独有指标的**定义与计算逻辑** | Active Zone Minutes (AZM) 是 Fitbit 基于心率区间的独有指标 → 映射为 Intensity Activity Minutes（强度活动分钟数），语义等价的通用概念 |
| **健康评估模型** | 理解竞品评估模型含义，找到**等价概念** | Readiness Score 是 Fitbit 综合 HRV/睡眠/活动的恢复评估 → Recovery Status（恢复状态），华为使用"恢复时间""跑力指数"等相近概念 |
| **隐含场景语境** | LLM 辅助识别肉眼难以发现的隐含竞品关联 | "Fitbit 追踪睡眠数据的具体算法" → 改写为华为 Watch 睡眠监测场景，而非简单替换品牌名 |

### 3.3 适配原则

1. **语义优先**: 理解 query 背后的用户意图，找到华为生态中语义等价的表述
2. **指标溯源**: 竞品专有指标需追溯其定义（如 AZM 的心率区间计算逻辑），再选最接近的华为/通用概念
3. **场景自洽**: 改写后的 query 在华为 Watch + Huawei Health 场景下逻辑成立
4. **LLM 辅助发现**: 批量扫描识别非直接命名但与 Google 生态相关的隐含场景

### 3.4 效果

- 4,170 条 → 7,424 条（含扩展）
- 5 个维度系统性适配，非机械文本替换

## 4. UMAP+HDBSCAN 聚类参数调优优化

### 4.1 优化背景

本人在团队提供的聚类框架基础上，负责参数的系统性调优。目标是在**噪声率、簇数量、簇粒度**三者之间找到最优平衡。

### 4.2 调优全过程（9 轮实验）

#### 阶段一：初始基线 → 噪声过多

```bash
uv run hqb cluster --umap-n-neighbors 15 --umap-n-components 64 \
  --umap-min-dist 0.0 --hdbscan-min-cluster-size 5 \
  --hdbscan-min-samples 3 --hdbscan-epsilon 0.03 --hdbscan-method eom --parallel 7
# → 2,054 clusters, 17,859 noise (~30%)
```

#### 阶段二：降噪探索 → 发现关键参数

| 实验 | 关键调整 | 噪声率 | 发现 |
|------|---------|--------|------|
| 方案A | eps↑0.06, min_samples↓1 | ~16% | **min_samples=1 是降噪关键** |
| 方案B | eps↑0.08, min_samples↑2 | ~20% | 保守参数反而噪声升高 |

#### 阶段三：高维方案收敛 → v1

```bash
uv run hqb cluster --umap-n-neighbors 15 --umap-n-components 64 \
  --umap-min-dist 0.0 --hdbscan-min-cluster-size 3 \
  --hdbscan-min-samples 1 --hdbscan-epsilon 0.08 --hdbscan-method eom --parallel 7
# → 1,899 clusters, 4,843 noise (~8%) ← v1 推荐
```

#### 阶段四：低维度探索

| 实验 | n_comp | min_samp | eps | 簇数 | 噪声率 | 发现 |
|------|--------|----------|-----|------|--------|------|
| — | 12 | 6 | 0.18 | 365 | ~17% | 簇过少 |
| **v2** | 12 | 1 | 0.18 | 883 | ~5% | 噪声最低但大簇需细分 |
| — | 12 | 1 | 0.08 | 3,516 | ~17% | eps 过小 |
| **v3** | 12 (min_dist=0.0) | 1 | 0.08 | 1,823 | ~8% | 低维均衡方案 |

### 4.3 参数影响规律

| 参数 | 降低 → | 增加 → |
|------|--------|--------|
| epsilon | 更多噪声 | 更少噪声 |
| min_samples | 更少噪声 | 更多噪声 |
| min_cluster_size | 更少噪声 | 更多噪声 |
| min_dist | 点聚集，边界模糊 | 点分散，识别清晰 |
| n_components | 更少簇(粗粒度) | 更多簇(细粒度) |

### 4.4 聚类准确率验证

| 指标 | 值 |
|------|---|
| 总样本数 | 6,734 |
| 总合理数 | 4,523 |
| **全局准确率** | **67.2%** |

### 4.5 调优成果

| 指标 | 优化前 | 优化后 | 改进 |
|------|--------|--------|------|
| 噪声率 | ~30% | ~8% | **-73%** |
| 全局准确率 | — | 67.2% | 基线建立 |

## 5. 统一审查框架优化

### 5.1 优化背景

面对 49,200+ 条生成数据 + 5 个独立 Domain，传统人工逐条审查完全不可行。

### 5.2 Checker 自适应框架设计

**核心优化**: Checker 自动从 `_DOMAIN_PROMPT_MAP` 读取任意 Domain 的 Prompt 约束，注入审查 prompt：

- **零配置审查**: 任意 Domain 注册 Prompt 后 Checker 即可审查
- **约束自动同步**: 审查标准与生成 Prompt 约束保持一致
- **抽样定位**: 100 条抽样即定位主要问题，等效数小时人工审查
- **结构化报告**: 按 Query 类型 / 知识主题 / 问题类型三维聚合

### 5.3 审查效果验证

| 维度 | 结果 |
|------|------|
| 总体正常率 | 81.0% (general_medical 抽样) |
| 最高频问题 | 语言不自然 (47.4%) → 指导术语控制 |
| 次高频问题 | 类型不匹配 (31.6%) → 指导类型边界 |
| 最高风险主题 | 运动损伤预防与康复 (26.7%) → 定向优化 |
| 最稳定类型 | 知识计算 Query (9.1%) → 作为标杆 |

### 5.4 流程价值

- **新 Domain 审查成本**: 从"编写审查规则"降至**零代码**
- **审查效率**: 100 条/数分钟 vs 人工数小时
- **优化闭环**: 审查→发现问题→Prompt 修改→重新生成→再审查

## 6. 架构可扩展性优化

### 6.1 Domain 可插拔架构

本系统最核心的架构优化是 **Domain 可插拔设计**——新增任何数据维度的边际成本趋近于零：

```
新 Domain 接入（零代码扩展）：
  1. prompts/ 新建 prompt 文件
  2. output_db.py 注册表 DDL
  3. generator.py 注册映射
  4. Checker 自动适配
──────────────────────────────
全链路自动打通: 生成 → 聚类 → 审查
```

**已验证**: 5 个 Domain（general_medical / health / sports / sports_health / red_team）均通过此流程接入，每个 Domain 拥有独立 Prompt 文件，各约 500 条，可持续生成。

### 6.2 系统工程保障

| 优化项 | 方案 | 效果 |
|--------|------|------|
| 并发生成 | asyncio + semaphore | 支持 49,200 条大批量 |
| 断点续传 | checkpoint 机制 | 中断后 <5s 恢复 |
| 指数退避 | 2s→30s cap | API 限流自动恢复 |
| 失败记录 | failed_tasks 表 | 异常完整可溯 |

## 7. 优化效果总结

### 7.1 关键指标

| 指标 | 初始/优化前 | 最终/优化后 | 改进 |
|------|-----------|-----------|------|
| 聚类噪声率 | ~30% | ~8% | **-73%** |
| 聚类全局准确率 | — | 67.2% | 基线建立 |
| LLM 去重损失率 | — | 0.62% | 极低 |
| 运动类别覆盖 | 单项 160 | 单项+组合 1,043 | **+552%** |
| 健康类别覆盖 | 单特征 13 | 单+双+三 377 | **+2,800%** |
| 新 Domain 接入成本 | 数百行代码 | 1 个 prompt + 2 行注册 | **趋近于零** |
| 新 Domain 审查成本 | 编写审查规则 | 零代码自适应 | **-100%** |
| Checker 审查正常率 | — | 81% (general_medical) | 首轮即定位 |

### 7.2 优化方法论

1. **Prompt 工程前置**: 三维度类型 + 字段硬约束 + 组合数学，从生成阶段最大化保证质量
2. **多维度场景适配**: 谷歌数据不做机械替换，从品牌/AI能力/指标/模型/场景 5 个维度语义等价适配
3. **系统性参数调优**: 9 轮 HDBSCAN 实验，建立参数-效果定量关系，科学收敛最优方案
4. **自动化审查闭环**: Checker 自适应 + 抽样，数分钟完成等效数小时人工审查，直接输出优化方向
5. **可插拔架构设计**: Domain 维度扩展的边际成本趋近于零，支持生态系统持续迭代
