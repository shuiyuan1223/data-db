## 2. 数据合成 Prompt 优化

### 2.1 优化项一：三维度 Query 类型体系设计

**问题**: 传统单一类型的 query 生成无法覆盖不同个性化层次的用户需求。

**优化方案**:
- 设计 **通用 / 显性个性化 / 隐性个性化** 三维度体系：
  - **通用 (10条)**: 不涉及个人数据的通用健康查询
  - **显性个性化 (10条)**: 直接意图涉及个人数据（如 "我的血压数据显示..."）
  - **隐性个性化 (10条)**: 隐含意图涉及个人数据（如 "最近运动后总觉得累..."）
- 每个类别严格 10+10+10 = 30 条

**效果**:
- 49,200 条 query 类型分布完全均衡（通用:显性:隐性 = 1:1:1）
- 去重率仅 0.62%（305 条），说明三维度设计有效避免了类型间的语义重叠

### 2.2 优化项二：运动特征的 Sparse 组合生成策略

**问题**: 仅覆盖单项运动不足以反映真实用户多运动场景。

**优化方案**:
- **单项 + 跨项组合** 的 Sparse 组合（非完全排列）：
  - 单项: 160 个独立运动项目（15.3%）
  - 跨项: 883 个基于运动科学类间关系的合理组合（84.7%）
- 组合逻辑示例：有氧+力量（跑步+力量训练）、球类跨项（篮球+足球）

**效果**: 1,043 标签 × 30 条 = 31,290 条运动 query，覆盖真实用户的多运动场景

### 2.3 优化项三：健康特征的组合数学设计

**问题**: 单指标 query 无法覆盖多指标关联的复杂健康查询场景。

**优化方案**:
- 13 个基础健康特征的数学组合：
  - C(13,1) = 13 单特征 → 390 条
  - C(13,2) = 78 双特征 → 2,340 条
  - C(13,3) = 286 三特征 → 8,580 条
- 合计 377 标签 × 30 条 = 11,310 条

**效果**: 双/三特征组合占比 96.5%，充分覆盖复杂多指标关联查询

### 2.4 优化项四：数据字段目录硬约束

**问题**: LLM 容易产生"字段幻觉"——输出不在预定义目录中的虚假健康数据字段。

**优化方案**:
- 在 `general_medical.py` 中定义 100+ 数据字段目录（DATA_FIELD_CATALOG）
- 将字段目录嵌入 SYSTEM_PROMPT，硬约束 LLM 只能从目录中选取

**效果**: Checker "格式违规" 缺陷率显著下降

### 2.5 优化项五：Red Team 多攻击维度 + 可扩展设计

**问题**: 红队测试需覆盖多种攻击模式，且未来可能增加新攻击类型。

**优化方案**:
- 4 类独立攻击 prompt（prompt_injection / persona_induction / function_inquiry / active_clarification）
- 每类攻击有独立的 SYSTEM_PROMPT，通过 query_category 字段区分
- **可扩展性**: 新增攻击类型只需在 `red_team.py` 中添加新 category 的 prompt，无需修改框架

**效果**: 约 500 条红队样本，4 类攻击全覆盖，可按需持续生成和扩展新攻击维度

### 2.6 优化项六：Domain 可插拔架构

**问题**: 每新增一个数据维度（Domain），需大量重复开发生成+审查代码。

**优化方案**:
```
新 Domain 接入流程：
  1. prompts/ 新建 prompt 文件 → 实现 build_messages()
  2. output_db.py 注册表 DDL → _DOMAIN_TABLE_MAP
  3. generator.py 注册 → _DOMAIN_PROMPT_MAP
  4. Checker 自动适配 → 无需额外代码
```

**效果**:
- 5 个 Domain 已通过此流程验证
- 新 Domain 接入成本从"数百行代码+调试"降至"1 个 prompt 文件+2 行注册"
- 数据维度扩展的边际成本趋近于零

## 3. 谷歌数据集多维度场景适配优化

### 3.1 优化背景

谷歌数据集原始 query 深度嵌入 Google/Fitbit 生态，**不能简单做关键词替换**，需要从多个维度理解竞品生态的语境，再适配到华为 Watch + Huawei Health 的产品场景中。

### 3.2 多维度审视角度

| 维度 | 审视角度 | 分析与适配 |
|------|---------|-----------|
| **品牌生态绑定** | 识别 query 中与 Google/Fitbit 产品直接绑定的表述 | Fitbit → Huawei Watch。不仅替换品牌名，更要确保改写后的 query 在华为产品能力范围内逻辑自洽 |
| **AI 功能命名** | 区分竞品的 AI 功能名称与华为对应功能的差异 | Fitbit LLM → Huawei Assistant。需理解两者功能边界的差异，避免生成华为 Assistant 不支持的交互场景 |
| **专有指标体系** | 深入分析竞品独有健康指标的**定义与计算逻辑**，找到语义最接近的华为/通用指标 | Active Zone Minutes (AZM) 是 Fitbit 基于心率区间计算的独有指标 → 映射为 Intensity Activity Minutes（强度活动分钟数），华为及通用的中高强度活动时间概念 |
| **健康评估模型** | 理解竞品特有的健康评估模型的含义，找到华为生态的**等价概念** | Readiness Score 是 Fitbit 综合 HRV/睡眠/活动的恢复评估 → Recovery Status（恢复状态），华为使用"恢复时间""跑力指数"等相近概念 |
| **隐含场景语境** | 通过 LLM 辅助识别肉眼难以发现的隐含竞品关联 | "Fitbit 用于追踪睡眠数据的具体算法" → 需改写为华为 Watch 的睡眠监测场景，而非简单替换品牌名 |

### 3.3 适配原则

1. **语义优先**: 理解 query 背后的用户意图，找到华为生态中语义等价的表述，而非文本层面的机械替换
2. **指标溯源**: 竞品专有指标需追溯其定义（如 AZM 的心率区间计算逻辑），再选择最接近的华为/通用概念
3. **场景自洽**: 改写后的 query 必须在华为 Watch + Huawei Health 场景下逻辑成立
4. **LLM 辅助发现**: 通过 LLM 批量扫描识别非直接命名但与 Google 生态相关的隐含场景

### 3.4 优化效果

- 原始 4,170 条 → 适配后 7,424 条
- 涵盖品牌/AI 能力/指标/模型/场景 5 个维度的系统性适配
- 改写后数据可直接用于华为 Watch 场景的模型训练，无竞品术语残留

## 4. UMAP+HDBSCAN 聚类参数调优优化（核心优化）

### 4.1 优化背景

本人负责对谷歌数据集 + 现网抽样数据的聚类参数进行系统性调优。聚类框架由团队实现，本人在此基础上通过 9 轮实验探索 6 个参数维度的最优组合，将噪声率从 ~30% 降至 ~8%。

### 4.2 调优全过程

#### 第一阶段：初始基线（噪声过多）

```bash
uv run hqb cluster --umap-n-neighbors 15 --umap-n-components 64 --umap-min-dist 0.0 \
  --hdbscan-min-cluster-size 5 --hdbscan-min-samples 3 --hdbscan-epsilon 0.03 \
  --hdbscan-method eom --parallel 7
# 结果：2,054 clusters, 17,859 noise points (~30%)
```

**问题**: 噪声率 ~30%，大量样本未被聚类，数据利用率低。

#### 第二阶段：降噪探索

| 实验 | 关键调整 | 簇数 | 噪声率 | 发现 |
|------|---------|------|--------|------|
| 方案 A | eps↑0.06, min_samples↓1 | 3,595 | ~16% | **min_samples=1 是降噪关键** |
| 方案 B | eps↑0.08, min_samples↑2 | — | ~20% | 保守参数反而噪声升高 |
| — | min_cluster=5, min_samples=1 | 1,765 | ~18% | min_cluster 过大 |
| — | min_cluster=3, min_samples=1 | 3,069 | ~14% | 簇数偏多 |

#### 第三阶段：高维方案收敛 → v1

```bash
uv run hqb cluster --umap-n-neighbors 15 --umap-n-components 64 --umap-min-dist 0.0 \
  --hdbscan-min-cluster-size 3 --hdbscan-min-samples 1 --hdbscan-epsilon 0.08 \
  --hdbscan-method eom --parallel 7
# 结果：1,899 clusters, 4,843 noise points (~8%) ← v1 推荐方案
```

**关键参数组合**: `min_dist=0.0` + `min_samples=1` + `epsilon=0.08` 达到最佳平衡。

#### 第四阶段：低维度探索

尝试 UMAP n_components 从 64 降至 12，探索粗粒度方案：

| 实验 | min_cluster | min_samples | epsilon | 簇数 | 噪声率 | 发现 |
|------|-----------|------------|---------|------|--------|------|
| — | 8 | 6 | 0.18 | 365 | ~17% | 簇过少 |
| — | 3 | 6 | 0.18 | 455 | ~17% | 仍过少 |
| **v2** | **3** | **1** | **0.18** | **883** | **~5%** | **噪声最低，但大簇需细分** |
| — | 3 | 1 | 0.08 | 3,516 | ~17% | eps 过小 |
| **v3** | **3** | **1** (min_dist=0.0) | **0.08** | **1,823** | **~8%** | **低维均衡方案** |

### 4.3 参数影响规律总结

| 参数 | 降低 → 效果 | 增加 → 效果 |
|------|------------|------------|
| **epsilon** | 更多噪声 ⚠️ | 更少噪声 ✅ |
| **min_samples** | 更少噪声 ✅ | 更多噪声 ⚠️ |
| **min_cluster_size** | 更少噪声 ✅ | 更多噪声 ⚠️ |
| **min_dist** | 点聚集，边界模糊 | 点分散，识别更清晰 |
| **n_components** | 更少簇(粗粒度)，大簇多 | 更多簇(细粒度) |

### 4.4 最终方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **v1** (64维) | 1,899 簇粒度适中，~8% 噪声 | — | 细粒度分析 |
| v2 (12维) | 最低噪声 5%，883 簇 | 大簇需细分 | 粗粒度+二次聚类 |
| **v3** (12维) | 1,823 簇均衡，~8% 噪声 | — | 通用方案 |

### 4.5 聚类准确率验证

| 指标 | 值 |
|------|---|
| 总样本数 | 6,734 |
| 总合理数 | 4,523 |
| **全局准确率** | **67.2%** |
| 高准确率簇 (≥90%) | 睡眠/步数/中医等领域 |
| 低准确率簇 (<30%) | 9 个簇，主因为主题混杂 |

### 4.6 调优成果

| 指标 | 优化前 | 优化后 | 改进幅度 |
|------|--------|--------|---------|
| 噪声率 | ~30% | ~8% | **-73%** |
| 簇数量 | 2,054 (过多) | 1,823-1,899 (适中) | 合理化 |
| 全局准确率 | — | 67.2% | 基线建立 |

## 5. Checker 审查流程优化

### 5.1 优化背景

传统的数据质量审查依赖人工逐条检查，面对 49,200+ 条数据量完全不可行。

### 5.2 统一审查框架设计

**核心优化**: Checker 自动从 `_DOMAIN_PROMPT_MAP` 读取生成 prompt 的约束条件，注入审查 prompt，实现：
- **零配置审查**: 新 Domain 注册 prompt 后 Checker 即可审查，无需编写审查规则
- **约束同步**: 审查标准自动与生成约束保持一致，消除人工同步错误
- **抽样定位**: 100 条抽样即可定位主要问题类型，等效数小时人工审查

### 5.3 审查效果验证（general_medical 抽样 100 条）

| 维度 | 结果 | 发现 |
|------|------|------|
| 总体正常率 | 81.0% | 首轮即建立质量基线 |
| 最高频问题 | 语言不自然 (47.4%) | → 指导 Prompt 增加术语层次控制 |
| 次高频问题 | 类型不匹配 (31.6%) | → 指导 Prompt 强化类型边界定义 |
| 最高风险主题 | 运动损伤预防与康复 (26.7%) | → 定向优化该主题 Prompt |
| 最低风险类型 | 知识计算 Query (9.1%) | → 该类型 Prompt 设计可作为标杆 |

### 5.4 流程价值量化

| 对比项 | 传统人工审查 | Checker 自动审查 |
|--------|-----------|----------------|
| 审查 100 条耗时 | 数小时 | 数分钟 |
| 新 Domain 接入成本 | 编写专用审查规则 | 零代码 |
| 审查标准一致性 | 依赖审查人员经验 | 自动与生成约束同步 |
| 报告结构化程度 | 自由文本 | 三维度自动聚合统计 |
| 优化方向指引 | 模糊 | 直接映射到 Prompt 具体优化点 |

## 6. 优化效果总结

### 6.1 关键指标改进

| 指标 | 初始/优化前 | 最终/优化后 | 改进 |
|------|-----------|-----------|------|
| 聚类噪声率 | ~30% | ~8% | **-73%** |
| 聚类全局准确率 | — | 67.2% | 基线建立 |
| LLM 去重损失率 | — | 0.62% | 极低 |
| 运动类别覆盖 | 单项 160 | 单项+组合 1,043 | **+552%** |
| 健康类别覆盖 | 单特征 13 | 单+双+三 377 | **+2,800%** |
| 新 Domain 接入成本 | 数百行代码 | 1 个 prompt 文件 + 2 行注册 | **趋近于零** |
| 新 Domain 审查成本 | 编写审查规则 | 零代码自动适配 | **-100%** |
| 谷歌数据适配维度 | — | 5 个维度系统性适配 | 非机械替换 |
| Checker 抽样审查 general_medical | — | 81% 正常率 | 首轮即定位问题 |

### 6.2 优化方法论总结

1. **Prompt 约束前置**: 三维度类型体系 + 字段硬约束 + 组合数学设计，从生成阶段就最大化保证数据质量，而不是依赖后期清洗
2. **系统性参数调优**: 9 轮 HDBSCAN 实验，建立参数-效果的定量关系，科学收敛至最优方案
3. **多维度场景适配**: 谷歌数据改写不是关键词替换，而是从品牌/AI能力/指标/模型/场景 5 个维度理解竞品生态后进行语义等价适配
4. **自动化审查闭环**: Checker 自适应 + 抽样策略，数分钟完成等效数小时人工审查，直接输出优化方向
5. **架构可扩展性**: Domain 可插拔设计，数据维度的扩展边际成本趋近于零，支持持续迭代
