## 1. 测试概述

### 1.1 系统定位

本人围绕已有的健康查询基准数据库，构建了一套**端到端 AI 数据自动化生态系统**，核心能力是：

1. **Prompt 驱动的多 Domain 数据合成**: 5 个独立 Domain，各有专属 Prompt 模板，各约 500 条，可持续生成扩展
2. **多源数据清洗与场景适配**: 谷歌数据 5 维度生态改写 + 现网数据分类抽样
3. **统一质量审查闭环**: Checker 自适应任意 Domain，抽样即定位问题，反馈 Prompt 优化

### 1.2 数据成果总览

| 数据来源 | 数量 | 特性 |
|----------|------|------|
| LLM 自动合成（3 大板块 + 5 个 Domain） | 49,200 + 各 Domain ~500 | 结构化，三维度类型体系 |
| 谷歌数据场景适配 | 4,170→7,424 | 5 维度生态改写 |
| 现网数据分类抽样 | 589,289→4,111 | 真实长尾数据 |
| **合计** | **56,319** (去重后) | **多源异构** |

## 2. 多 Domain Prompt 生成性能

### 2.1 Domain 可插拔架构

本系统的每个 Domain 都拥有**独立的 Prompt 模板文件**，是独立的数据生成维度：

| Domain | Prompt 文件 | 人设来源 | 规模 | 特色能力 |
|--------|-----------|---------|------|---------|
| general_medical | `prompts/general_medical.py` | benchmark.db (domain 过滤) | ~500 条 | 100+ 数据字段目录硬约束 |
| health | `prompts/health.py` | benchmark.db (domain 过滤) | ~500 条 | topic_count 多子域主题支持 |
| sports | `prompts/sports.py` | LLM 内联生成 | ~500 条 | 无需外部 persona，LLM 自生成 |
| sports_health | `prompts/sports_health.py` | benchmark.db (全量) | ~500 条 | 跨域 persona 健康+运动融合 |
| red_team | `prompts/red_team.py` | 无人设 | ~500 条 | 4 类攻击（injection/induction/inquiry/clarification） |

**每个 Domain 均可独立持续生成**，增加 `--num-groups` 或 `--topics` 即可扩展数据量。

### 2.2 批量生成板块性能

除 5 个 Domain 外，系统还支持大规模批量生成（三大板块）：

| 板块 | 类别类型 | 标签数量 | 生成规则 | Query 总数 |
|------|---------|---------|---------|-----------|
| 知识目录 | 树状知识节点 | 220 | 30 条/类 | 6,600 |
| 运动 | 单项(160) + 跨项组合(883) | 1,043 | 30 条/类 | 31,290 |
| 健康 | C(13,1)+C(13,2)+C(13,3) | 377 | 30 条/类 | 11,310 |
| **合计** | | **1,640** | | **49,200** |

**去重效果**: 49,200 → 48,895（损失仅 0.62%），验证了三维度类型设计的去重有效性

### 2.3 三维度 Query 类型分布

每个类别严格生成 10 通用 + 10 显性个性化 + 10 隐性个性化：
- **通用**: 不涉及个人数据
- **显性个性化**: 直接意图涉及个人数据
- **隐性个性化**: 隐含意图涉及个人数据

分布均衡度 = 1:1:1（100% 严格均衡）

## 3. Checker 自动审查机制与闭环优化

> **说明**: 本章展示的审查结果为小样本抽样（100 条），目的不在于给出统计意义上的质量结论，而是**验证审查机制本身的有效性**——即 Checker 能否在极少样本下快速定位问题类型、高风险主题，并直接指导 Prompt 迭代优化。这套机制可零代码应用于任意 Domain、任意规模的数据。

### 3.1 审查机制的核心设计

Checker 是本生态系统的**质量闭环引擎**，其核心价值不在于审查多少条数据，而在于：

```
┌─────────────────────────────────────────────────────────┐
│                    Checker 闭环优化机制                    │
│                                                          │
│  ① 自动读取 Domain Prompt 约束 → 生成审查标准             │
│  ② 小样本抽样审查 → 快速定位问题类型和高风险主题          │
│  ③ 结构化报告（三维聚合）→ 直接映射 Prompt 优化方向       │
│  ④ 修改 Prompt → 重新生成 → 再审查 → 正常率持续提升       │
│                                                          │
│  关键特性：                                               │
│  · 任意 Domain 零代码接入（自动适配 Prompt 约束）          │
│  · 抽样即可定位（不依赖全量审查）                          │
│  · 审查标准自动与生成 Prompt 保持同步                      │
└─────────────────────────────────────────────────────────┘
```

### 3.2 机制效率对比

| 对比项 | 传统人工审查 | Checker 自动审查 |
|--------|-----------|----------------|
| 审查 100 条耗时 | 数小时 | 数分钟 |
| 新 Domain 接入 | 编写专用审查规则 | 零代码自适应 |
| 审查标准一致性 | 依赖审查人员经验 | 自动与生成 Prompt 约束同步 |
| 报告结构化程度 | 自由文本 | 三维度自动聚合（类型/主题/问题） |
| 优化方向指引 | 模糊 | 直接映射到 Prompt 具体修改点 |
| 可扩展性 | 人力线性增长 | 对任意 Domain/任意规模复用 |

### 3.3 机制验证：general_medical 抽样审查示例

以下为 general_medical Domain 的**小样本抽样审查示例**（100 条），用于验证审查机制的缺陷发现能力和优化指引能力。

**配置**: 随机抽样 100 条，分 4 批审查（batch_size=30），temperature=0.3

#### 总体结果

| 指标 | 值 |
|------|---|
| 抽样审查数 | 100 条（从全量中随机抽取） |
| 正常 Query | 81 条 (81.0%) |
| 标记有问题 | 19 条 (19.0%) |

#### 机制验证一：按 Query 类型定位问题分布

Checker 自动按 Query 类型维度聚合问题，快速定位哪类 Query 的 Prompt 需要优化：

| Query 类型 | 审查数 | 问题数 | 问题率 | 机制定位结论 |
|-----------|--------|--------|--------|------------|
| 专业知识 Query | 34 | 6 | 17.6% | 术语专业度需按 persona 背景控制 |
| 用户数据查询 Query | 33 | 9 | **27.3%** | **类型标注边界需在 Prompt 中强化** |
| 知识计算 Query | 33 | 3 | **9.1%** | 质量最稳定，Prompt 设计可作为标杆 |

#### 机制验证二：按知识主题定位高风险区域

Checker 自动按主题维度聚合，识别哪些知识主题的 Prompt 需优先优化：

| 知识主题 | 问题率 | 机制定位结论 |
|---------|--------|------------|
| 运动损伤预防与康复 | 4/15 (**26.7%**) | **最高风险，需优先优化该主题 Prompt** |
| 有氧训练与心肺功能提升 | 3/14 (21.4%) | 术语专业度偏高 |
| 运动营养与能量补给策略 | 3/15 (20.0%) | 部分 query 主题偏移 |
| 睡眠与运动恢复的关系 | 2/12 (16.7%) | 可接受 |
| 运动心率区间与训练强度管理 | 2/14 (14.3%) | 可接受 |
| 体脂管理与运动减脂效率 | 2/15 (13.3%) | 可接受 |
| 力量训练原则与肌肉恢复 | 2/15 (13.3%) | 可接受 |

#### 机制验证三：按问题类型 → 直接映射 Prompt 优化方向

| 问题类型 | 数量 | 占比 | Checker → Prompt 优化方向 |
|---------|------|------|-------------------------|
| 语言不自然 | 9 | **47.4%** | → Prompt 增加术语层次控制，与 persona 专业背景匹配 |
| 类型不匹配 | 6 | **31.6%** | → Prompt 强化三种 Query 类型的边界定义与判别示例 |
| 逻辑矛盾 | 3 | 15.8% | → Prompt 加强 query 核心意图与指定主题的锚定约束 |
| 重复/相似 | 1 | 5.3% | → Prompt 增加多样性约束 |

### 3.4 典型问题案例（审查机制发现）

**案例 1 — 语言不自然（机制最高频发现）**:
```
Query: "在运动生理学中，如何根据最大摄氧量百分比来精确划分训练强度区间？"
Checker 发现: 使用"最大摄氧量百分比""生理适应机制"等高度专业术语，
             若 persona 为普通用户则表述不自然。
→ 闭环优化: Prompt 约束术语复杂度与 persona 专业背景匹配
```

**案例 2 — 类型不匹配**:
```
Query: "帮我看看我过去一个月的每日平均心率和静息心率分别是多少？"
Checker 发现: 标注为知识计算 Query，实际为用户数据查询
→ 闭环优化: Prompt 为三种 Query 类型提供更清晰的分类边界
```

**案例 3 — 逻辑矛盾**:
```
主题: "运动营养与能量补给策略"
Query: "帮我看看上个月每次练完之后，手环记录的睡眠平均心率是多少？"
Checker 发现: query 内容是睡眠心率，与营养主题无关
→ 闭环优化: Prompt 强化 query 核心意图与指定主题的锚定
```

### 3.5 审查机制结论

| 结论 | 说明 |
|------|------|
| **小样本即有效** | 100 条抽样即清晰定位 3 类主要问题和 1 个高风险主题 |
| **问题→优化直接映射** | 每类问题直接对应 Prompt 的具体修改方向，无需人工分析 |
| **任意 Domain 可复现** | 同样机制即时应用于 health/sports/sports_health/red_team 等任意 Domain |
| **闭环持续优化** | 审查 → Prompt 修改 → 重新生成 → 再审查 → 正常率持续提升 |
| **人力成本极低** | 一次审查数分钟，替代数小时人工逐条检查 |

## 6. 结论

### 6.1 综合评价

| 维度 | 评价 | 等级 |
|------|------|------|
| 生态系统完整性 | 生成→审查全链路自动化闭环 | 优秀 |
| 架构可扩展性 | 5 个 Domain 各有独立 Prompt，新 Domain 零代码接入 | 优秀 |
| 审查机制有效性 | 小样本抽样即定位问题类型/高风险主题，直接映射 Prompt 优化方向 | 优秀 |
| 数据合成质量 | 去重率 0.62%，三维度类型 1:1:1 均衡 | 良好 |
| 审查效率 | 100 条/数分钟，替代数小时人工审查 | 优秀 |
| 场景适配 | 5 维度生态改写，非机械替换 | 良好 |

### 6.2 闭环优化路径

基于 Checker 机制已定位的问题，后续 Prompt 迭代方向：

1. **术语层次控制** → 语言不自然(47.4%)问题下降 → 正常率提升
2. **类型边界强化** → 类型不匹配(31.6%)问题下降
3. **主题锚定加强** → 逻辑矛盾(15.8%)问题下降
4. **各 Domain 全覆盖审查** → 利用同一机制建立全局质量基线
