## 1. 测试概述

### 1.1 系统定位

本人围绕已有的健康查询基准数据库，构建了一套**端到端 AI 数据自动化生态系统**，核心能力是：

1. **Prompt 驱动的多 Domain 数据合成**: 5 个独立 Domain，各有专属 Prompt 模板（含字段目录/运动指标词典/攻击模板等），各约 500 条，可持续生成扩展
2. **多源数据清洗与场景适配**: 谷歌数据 5 维度生态改写 + 现网 589,289 条分类抽样
3. **统一质量审查闭环**: Checker 从 Domain Prompt 中自动提取约束注入审查，抽样即定位问题，反馈 Prompt 优化

### 1.2 数据成果总览

## 2. 多 Domain Prompt 生成性能

### 2.1 Domain 可插拔架构与 Prompt 实现细节

每个 Domain 的 Prompt 文件实现两个核心函数：
- `build_persona_block(persona)` — 将 persona 字典渲染为 LLM 可读的结构化卡片（年龄/性别/BMI/健康关注/运动偏好等）
- `build_messages(persona, catalog_topic)` — 构造 [system_prompt, user_message] 消息对

| Domain | Prompt 文件 | 字段/指标体系 | 核心 Prompt 设计 |
|--------|-----------|-------------|----------------|
| general_medical | `general_medical.py` | 101 个字段（8 大类：基础体征/体成分/每日汇总/心率压力/睡眠/健康档案/营养/高频时序） | Prompt 要求 LLM 根据 persona 医学知识水平适配术语 |
| health | `health.py` | 52 个字段 + HEALTH_DATA_MARKDOWN 健康指标词典 | 支持下划线分隔的多域组合主题（如"心脏健康_睡眠健康"） |
| sports | `sports.py` | SPORTS_SPECIFIC_INDICATORS：30+ 运动专属指标词典 | REQUIRES_PERSONA=False，LLM 自行生成 15 个多样化内联 persona |
| sports_health | `sports_health.py` | 融合运动(preferred_sports/mastery/goal)与健康属性 | 跨域 persona：同时携带运动偏好和健康关注 |
| red_team | `red_team.py` | 4 套独立攻击模板 | injection(含难度分级)/induction(6种诱导技术)/inquiry(6维功能探测)/clarification(歧义类型) |

**运动指标词典示例（SPORTS_SPECIFIC_INDICATORS）**：
- 跑步类：配速/步频/步幅/触地时间/垂直振幅/左右平衡/跑步分析
- 骑行类：FTP/功率/踏频/坡度分析
- 游泳类：划水频率/SWOLF/泳距
- 高尔夫：挥杆速度/挥杆节奏/推杆数/开球准确率/差点

### 2.2 批量生成板块性能

| 板块 | 类别类型 | 标签数量 | 生成规则 | Query 总数 |
|------|---------|---------|---------|-----------|
| 知识目录 | 220 个末级树状知识节点 | 220 | 30 条/类（10通用+10显性+10隐性） | 6,600 |
| 运动 | 单项(160) + 跨项 Sparse 组合(883) | 1,043 | 30 条/类 | 31,290 |
| 健康 | 13 基础特征的 C(13,1)+C(13,2)+C(13,3) | 377 | 30 条/类 | 11,310 |
| **合计** | | **1,640** | | **49,200** |

**去重效果**: 49,200 → 48,895（去重率 0.62%），说明三维度类型设计有效避免了语义重叠。

### 2.3 三维度 Query 类型定义

| 类型 | 定义 | data_fields 要求 | 示例意图 |
|------|------|----------------|---------|
| Type 1 通用 | 不涉及个人数据的知识查询 | 无 | "高血压的诊断标准是什么" |
| Type 2 显性个性化 | 直接请求查看个人数据 | 1-3 个字段 | "帮我看看我上周的平均心率" |
| Type 3 隐性个性化 | 隐含需要个人数据做分析计算 | 2-4 个字段 | "最近运动后总觉得累，是不是训练过度了" |

分布均衡度 = 1:1:1（每类别严格 10+10+10）

### 2.4 生成引擎实现细节

generator.py 的任务编排流程：

```
_load_topics() → _build_task_list() → 按 persona×topic round-robin 排列
    ↓
asyncio.Semaphore(batch) 控制并发
    ↓
_check_is_generated() → 已存在则跳过（checkpoint）
    ↓
llm_client.chat_json(messages, temp=0.8) → LLM 调用
    ↓
_dispatch_insert() → 路由到对应 Domain 的 insert 函数
    ↓
失败 → _record_failure() → failed_tasks 表
```

**重试机制**: `delay = min(2 × 2^(retry-1), 30)` 秒，默认最大 100 次重试，API 超时 120s。

## 3. Checker 自动审查机制与闭环优化

> **说明**: 本章展示的审查结果为小样本抽样（100 条），目的不在于给出统计意义上的质量结论，而是**验证审查机制本身的有效性**——即 Checker 能否在极少样本下快速定位问题类型、高风险主题，并直接指导 Prompt 迭代优化。这套机制可零代码应用于任意 Domain、任意规模的数据。

### 3.1 审查机制的核心设计

Checker（`checker.py`）的工作流：

```
_extract_domain_info(domain)    ← 自动提取 SYSTEM_PROMPT + has_persona
        ↓
_load_queries(db, domain, ...)  ← 加载 queries（支持 persona 过滤 + 随机抽样）
        ↓
_build_review_prompt(info, batch) ← 将生成约束注入审查 prompt
        ↓
llm_client.chat_json(temp=0.3)  ← 低方差审查
        ↓
_write_report()                 ← 三维聚合报告
```

**6 维度缺陷分类体系**：

| 缺陷维度 | 检测内容 |
|---------|---------|
| 语言不自然 | 过于书面化/专业化，不符合 persona 的表达习惯 |
| 类型不匹配 | query 实际意图与标注的 query_type 不一致 |
| 逻辑矛盾 | query 内容与指定的知识主题或 persona 属性矛盾 |
| 格式违规 | 含禁止符号（带解释的括号、中英混杂等） |
| 内容重复 | 同 persona×topic 下出现句式/意图高度相似的 query |
| 质量低下 | intent 或 data_fields 与 query 内容不匹配 |

### 3.2 机制效率对比

| 对比项 | 传统人工审查 | Checker 自动审查 |
|--------|-----------|----------------|
| 审查 100 条耗时 | 数小时 | 数分钟 |
| 新 Domain 接入 | 编写专用审查规则 | `_extract_domain_info` 自动适配 |
| 审查标准一致性 | 依赖审查人员经验 | 自动与生成 Prompt 约束同步 |
| 报告结构化程度 | 自由文本 | 三维度自动聚合（类型/主题/问题） |
| 优化方向指引 | 模糊 | 直接映射到 Prompt 具体修改点 |
| 可扩展性 | 人力线性增长 | 对任意 Domain/任意规模复用 |

### 3.3 机制验证：general_medical 抽样审查示例

以下为 general_medical Domain 的**小样本抽样审查示例**（100 条），用于验证审查机制的缺陷发现能力和优化指引能力。

**配置**: `--domain general_medical --sample 100`，分 4 批审查（batch_size=30），temperature=0.3

#### 总体结果

| 指标 | 值 |
|------|---|
| 抽样审查数 | 100 条（从全量中随机抽取） |
| 正常 Query | 81 条 (81.0%) |
| 标记有问题 | 19 条 (19.0%) |

#### 机制验证一：按 Query 类型定位问题分布

| Query 类型 | 审查数 | 问题数 | 问题率 | 机制定位结论 |
|-----------|--------|--------|--------|------------|
| 专业知识 Query | 34 | 6 | 17.6% | 术语专业度需按 persona 背景控制 |
| 用户数据查询 Query | 33 | 9 | **27.3%** | **类型标注边界需在 Prompt 中强化** |
| 知识计算 Query | 33 | 3 | **9.1%** | 质量最稳定，Prompt 设计可作为标杆 |

#### 机制验证二：按知识主题定位高风险区域

| 知识主题 | 问题率 | 机制定位结论 |
|---------|--------|------------|
| 运动损伤预防与康复 | 4/15 (**26.7%**) | **最高风险，需优先优化该主题 Prompt** |
| 有氧训练与心肺功能提升 | 3/14 (21.4%) | 术语专业度偏高 |
| 运动营养与能量补给策略 | 3/15 (20.0%) | 部分 query 主题偏移 |
| 睡眠与运动恢复的关系 | 2/12 (16.7%) | 可接受 |
| 运动心率区间与训练强度管理 | 2/14 (14.3%) | 可接受 |
| 体脂管理与运动减脂效率 | 2/15 (13.3%) | 可接受 |
| 力量训练原则与肌肉恢复 | 2/15 (13.3%) | 可接受 |

#### 机制验证三：按问题类型 → 直接映射 Prompt 优化方向

| 问题类型 | 数量 | 占比 | Checker → Prompt 优化方向 |
|---------|------|------|-------------------------|
| 语言不自然 | 9 | **47.4%** | → Prompt 增加术语层次控制，与 persona 专业背景匹配 |
| 类型不匹配 | 6 | **31.6%** | → Prompt 强化三种 Query 类型的边界定义与判别示例 |
| 逻辑矛盾 | 3 | 15.8% | → Prompt 加强 query 核心意图与指定主题的锚定约束 |
| 重复/相似 | 1 | 5.3% | → Prompt 增加多样性约束 |

### 3.4 典型问题案例（审查机制发现）

**案例 1 — 语言不自然（最高频）**:
```
ID: 163 | persona: 136 | 主题: 运动心率区间与训练强度管理
Query: "在运动生理学中，如何根据最大摄氧量百分比来精确划分训练强度区间？"
Checker: 使用"最大摄氧量百分比""生理适应机制"等高度专业术语，
         若 persona 为普通用户则不自然。
→ 闭环优化: Prompt 中约束"普通用户使用日常语言，医护人员可使用专业术语"
```

**案例 2 — 类型不匹配**:
```
ID: 170 | persona: 161 | 主题: 运动损伤预防与康复
Query: "帮我看看我过去一个月的每日平均心率和静息心率分别是多少？"
Checker: 实际为用户数据查询(Type 2)，标注为知识计算(Type 3)
→ 闭环优化: Prompt 为三种 Query 类型提供更清晰的边界——Type 2 是"查看数据"，Type 3 是"基于数据做分析/计算/建议"
```

**案例 3 — 逻辑矛盾**:
```
ID: 143 | persona: 98 | 主题: 运动营养与能量补给策略
Query: "帮我看看上个月每次练完之后，手环记录的睡眠平均心率是多少？"
Checker: query 内容是睡眠心率，与营养主题无关
→ 闭环优化: Prompt 强化"query 核心意图必须与指定 catalog_topic 直接相关"
```

### 3.5 审查机制结论

| 结论 | 说明 |
|------|------|
| **小样本即有效** | 100 条抽样即清晰定位 3 类主要问题和 1 个高风险主题 |
| **问题→优化直接映射** | 每类问题直接对应 Prompt 的具体修改方向，无需人工分析 |
| **任意 Domain 可复现** | `_extract_domain_info` 自动提取，同样机制即时应用于任意 Domain |
| **闭环持续优化** | 审查 → Prompt 修改 → 重新生成 → 再审查 → 正常率持续提升 |
| **人力成本极低** | 一次审查数分钟，替代数小时人工逐条检查 |

## 6. 结论

### 6.1 综合评价

| 维度 | 评价 | 等级 |
|------|------|------|
| 生态系统完整性 | 生成→审查全链路自动化闭环，checkpoint+重试保障稳定性 | 优秀 |
| 架构可扩展性 | 5 个 Domain 各有独立 Prompt（含字段目录/指标词典/攻击模板），新 Domain 零代码接入 | 优秀 |
| 审查机制有效性 | 6 维度缺陷分类，小样本即定位问题，直接映射 Prompt 优化方向 | 优秀 |
| 数据合成质量 | 去重率 0.62%，三维度类型 1:1:1 均衡，101 字段目录硬约束 | 良好 |
| 审查效率 | 100 条/数分钟，`_extract_domain_info` 自动适配 | 优秀 |
| 场景适配 | 5 维度生态改写（品牌/AI/指标/模型/场景），非机械替换 | 良好 |

### 6.2 闭环优化路径

基于 Checker 机制已定位的问题，后续 Prompt 迭代方向：

1. **术语层次控制** → SYSTEM_PROMPT 增加"根据 persona 医学知识水平适配术语"约束 → 语言不自然(47.4%)下降
2. **类型边界强化** → 在 Prompt 中为 Type 1/2/3 提供更明确的判别示例 → 类型不匹配(31.6%)下降
3. **主题锚定加强** → 增加"query 核心意图必须与 catalog_topic 直接相关"约束 → 逻辑矛盾(15.8%)下降
4. **各 Domain 全覆盖审查** → 利用同一 Checker 机制对 health/sports/sports_health/red_team 执行审查 → 建立全局质量基线
