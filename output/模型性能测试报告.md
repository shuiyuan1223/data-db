## 1. 测试概述

### 1.1 系统定位

本人围绕已有的健康查询基准数据库，构建了一套**端到端 AI 数据自动化生态系统**，核心能力是：

1. **Prompt 驱动的多 Domain 数据合成**: 5 个独立 Domain，各有专属 Prompt 模板，各约 500 条，可持续生成扩展
2. **多源数据清洗与场景适配**: 谷歌数据 5 维度生态改写 + 现网数据分类抽样
3. **聚类参数系统性调优**: 9 轮 UMAP+HDBSCAN 实验，噪声率 30%→8%
4. **统一质量审查闭环**: Checker 自适应任意 Domain，抽样即定位问题，反馈 Prompt 优化

### 1.2 数据成果总览

| 数据来源 | 数量 | 特性 |
|----------|------|------|
| LLM 自动合成（3 大板块 + 5 个 Domain） | 49,200 + 各 Domain ~500 | 结构化，三维度类型体系 |
| 谷歌数据场景适配 | 4,170→7,424 | 5 维度生态改写 |

| **合计** | **56,319** (去重后) | **多源异构** |

## 2. 多 Domain Prompt 生成性能

### 2.1 Domain 可插拔架构

本系统的每个 Domain 都拥有**独立的 Prompt 模板文件**，是独立的数据生成维度：

| Domain | Prompt 文件 | 人设来源 | 规模 | 特色能力 |
|--------|-----------|---------|------|---------|
| general_medical | `prompts/general_medical.py` | benchmark.db (domain 过滤) | ~500 条 | 100+ 数据字段目录硬约束 |
| health | `prompts/health.py` | benchmark.db (domain 过滤) | ~500 条 | topic_count 多子域主题支持 |
| sports | `prompts/sports.py` | LLM 内联生成 | ~500 条 | 无需外部 persona，LLM 自生成 |
| sports_health | `prompts/sports_health.py` | benchmark.db (全量) | ~500 条 | 跨域 persona 健康+运动融合 |
| red_team | `prompts/red_team.py` | 无人设 | ~500 条 | 4 类攻击（injection/induction/inquiry/clarification） |

**每个 Domain 均可独立持续生成**，增加 `--num-groups` 或 `--topics` 即可扩展数据量。

### 2.2 批量生成板块性能

除 5 个 Domain 外，系统还支持大规模批量生成（三大板块）：

| 板块 | 类别类型 | 标签数量 | 生成规则 | Query 总数 |
|------|---------|---------|---------|-----------|
| 知识目录 | 树状知识节点 | 220 | 30 条/类 | 6,600 |
| 运动 | 单项(160) + 跨项组合(883) | 1,043 | 30 条/类 | 31,290 |
| 健康 | C(13,1)+C(13,2)+C(13,3) | 377 | 30 条/类 | 11,310 |
| **合计** | | **1,640** | | **49,200** |

**去重效果**: 49,200 → 48,895（损失仅 0.62%），验证了三维度类型设计的去重有效性

### 2.3 三维度 Query 类型分布

每个类别严格生成 10 通用 + 10 显性个性化 + 10 隐性个性化：
- **通用**: 不涉及个人数据
- **显性个性化**: 直接意图涉及个人数据
- **隐性个性化**: 隐含意图涉及个人数据

分布均衡度 = 1:1:1（100% 严格均衡）

## 3. Checker 自动审查性能

### 3.1 审查架构的流程价值

Checker 是本生态系统的**质量闭环核心**，其价值在于自动化与效率：

| 对比项 | 传统人工审查 | Checker 自动审查 |
|--------|-----------|----------------|
| 审查 100 条耗时 | 数小时 | 数分钟 |
| 新 Domain 接入 | 编写专用审查规则 | 零代码自适应 |
| 审查标准一致性 | 依赖审查人员经验 | 自动与生成 Prompt 约束同步 |
| 报告结构化 | 自由文本 | 三维度自动聚合 |
| 优化方向指引 | 模糊 | 直接映射到 Prompt 具体优化点 |

### 3.2 general_medical Domain 审查结果

**配置**: 随机抽样 100 条，分 4 批审查（batch_size=30），temperature=0.3

#### 总体结果

| 指标 | 值 |
|------|---|
| 审查总数 | 100 条（抽样） |
| 正常 Query | 81 条 (**81.0%**) |
| 标记有问题 | 19 条 (19.0%) |

#### 按 Query 类型的问题分布

| Query 类型 | 审查数 | 问题数 | 问题率 | 分析 |
|-----------|--------|--------|--------|------|
| 专业知识 Query | 34 | 6 | 17.6% | 部分术语过于专业化 |
| 用户数据查询 Query | 33 | 9 | **27.3%** | 类型标注与实际意图偶有不一致 |
| 知识计算 Query | 33 | 3 | **9.1%** | 质量最稳定，可作为标杆 |

#### 按知识主题的问题分布

| 知识主题 | 问题率 | 风险等级 |
|---------|--------|---------|
| 运动损伤预防与康复 | 4/15 (**26.7%**) | 高 → 需优先优化 Prompt |
| 有氧训练与心肺功能提升 | 3/14 (21.4%) | 中 |
| 运动营养与能量补给策略 | 3/15 (20.0%) | 中 |
| 睡眠与运动恢复的关系 | 2/12 (16.7%) | 低 |
| 运动心率区间与训练强度管理 | 2/14 (14.3%) | 低 |
| 体脂管理与运动减脂效率 | 2/15 (13.3%) | 低 |
| 力量训练原则与肌肉恢复 | 2/15 (13.3%) | 低 |

#### 按问题类型的分布

| 问题类型 | 数量 | 占比 | 典型表现 | Prompt 优化方向 |
|---------|------|------|---------|---------------|
| 语言不自然 | 9 | **47.4%** | HRV/VO2max/EPOC 等术语过于专业 | 约束术语层次与 persona 背景匹配 |
| 类型不匹配 | 6 | **31.6%** | 数据查询标注为知识 Query | 强化三种类型的边界定义与判别示例 |
| 逻辑矛盾 | 3 | 15.8% | query 内容与指定主题关联弱 | 加强主题锚定约束 |
| 重复/相似 | 1 | 5.3% | 不同 persona 生成句式相似 query | 增加多样性约束 |

### 3.3 典型问题案例

**案例 1 — 语言不自然（最高频）**:
```
Query: "在运动生理学中，如何根据最大摄氧量百分比来精确划分训练强度区间？"
问题: 使用"最大摄氧量百分比""生理适应机制"等高度专业术语，
      若 persona 为普通用户则表述不自然。
→ Prompt 优化: 约束术语复杂度与 persona 专业背景匹配
```

**案例 2 — 类型不匹配**:
```
Query: "帮我看看我过去一个月的每日平均心率和静息心率分别是多少？"
标注: 知识计算 Query | 实际: 用户数据查询
→ Prompt 优化: 为三种 Query 类型提供更清晰的分类边界
```

**案例 3 — 逻辑矛盾**:
```
主题: "运动营养与能量补给策略"
Query: "帮我看看上个月每次练完之后，手环记录的睡眠平均心率是多少？"
问题: query 内容是睡眠心率，与营养主题无关
→ Prompt 优化: 强化 query 核心意图与指定主题的锚定
```

### 3.4 审查流程结论

| 结论 | 说明 |
|------|------|
| 首轮即定位问题 | 100 条抽样 → 语言不自然(47%) / 类型不匹配(32%) / 逻辑矛盾(16%) |
| 直接指导优化 | 每类问题直接映射到 Prompt 具体修改方向 |
| 流程可复现 | 同样流程即时应用于任意 Domain，无需额外开发 |
| 优化闭环 | 审查 → 修改 Prompt → 重新生成 → 再审查 → 正常率持续提升 |

## 7. 结论

### 7.1 综合评价

| 维度 | 评价 | 等级 |
|------|------|------|
| 生态系统完整性 | 生成→聚类→审查全链路自动化闭环 | 优秀 |
| 架构可扩展性 | 5 个 Domain 各有独立 Prompt，新 Domain 零代码接入 | 优秀 |
| 数据合成质量 | 去重率 0.62%，Checker 抽样 81% 正常率 | 良好 |
| 审查效率 | 100 条/数分钟，等效数小时人工审查 | 优秀 |

### 7.2 优化方向

1. Prompt 术语层次控制 → 正常率 81%→90%+

3. 低准确率簇二次聚类 → 全局准确率 67%→72%+
4. 各 Domain Checker 审查全覆盖 → 建立全局质量基线
