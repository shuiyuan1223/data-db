## 2. LLM 数据合成性能报告

### 2.1 生成规模与结构

**生成规则**: 1 个类别生成 30 条 Query（10 条通用 + 10 条显性个性化 + 10 条隐性个性化）

| 数据板块 | 类别/标签类型 | 标签数量 | 单标签 Query 数 | 板块 Query 总数 |
|---------|-------------|---------|----------------|---------------|
| 知识目录 | 树状知识节点 | 220 | 30 | 6,600 |
| 运动 (Sports) | 单项与跨项组合 | 1,043 | 30 | 31,290 |
| 健康 (Health) | 组合特征 (C1,C2,C3) | 377 | 30 | 11,310 |
| **总计** | **全量数据** | **1,640** | — | **49,200** |

**去重效果**: 49,200 → 48,895（去重率仅 0.62%，说明三维度类型设计有效避免了语义重叠）

### 2.2 大 Domain 生成概况

除上述三大板块外，系统内置 5 个可独立扩展的大 Domain：

| Domain | 估计规模 | 人设来源 | 可扩展性 |
|--------|---------|---------|---------|
| general_medical | ~500 条 | benchmark.db (domain 过滤) | 可持续追加生成轮次 |
| health | ~500 条 | benchmark.db (domain 过滤) | 可持续追加生成轮次 |
| sports | ~500 条 | LLM 内联生成 | 可持续追加生成轮次 |
| sports_health | ~500 条 | benchmark.db (全量) | 可持续追加生成轮次 |
| red_team | ~500 条 | 无人设，4 类攻击 | 可扩展新攻击维度 |

**关键架构特性**: 每个 Domain 只需注册 prompt 模块即可接入生成 + 审查全链路，新增 Domain 的边际成本趋近于零。

## 3. Checker 自动审查测试结果

### 3.1 审查架构的核心价值

Checker 的设计目标是**以最小人力投入快速定位质量瓶颈**，而非追求全量人工逐条审查：
- 抽样 100 条即可定位主要问题类型和高风险知识主题
- 自动适配任意 Domain，无需人工编写审查规则
- 结构化报告按三个维度聚合，直接指导 Prompt 优化方向

### 3.2 general_medical 抽样审查结果

**审查配置**: 随机抽样 100 条，分 4 批审查（batch_size=30）

| 正常 Query | 81 条 (81.0%) |

### 3.3 典型问题案例分析

#### 案例 1：语言不自然（最高频问题）
```
ID: 163 | persona: 136 | 主题: 运动心率区间与训练强度管理
Query: 在运动生理学中，如何根据最大摄氧量百分比来精确划分训练强度区间，
       各区间对应的生理适应机制是什么？
问题: 使用了"最大摄氧量百分比""生理适应机制"等高度专业术语，
      若 persona 为普通用户则不自然。
```

**优化方向**: Prompt 中约束术语使用层次，要求与 persona 的专业背景匹配。

#### 案例 2：类型不匹配
```
ID: 170 | persona: 161 | 主题: 运动损伤预防与康复
Query: 帮我看看我过去一个月的每日平均心率和静息心率分别是多少？
标注类型: 知识计算Query
问题: 实际为用户数据查询，与标注类型不符。
```

**优化方向**: 在 Prompt 中为三种 Query 类型提供更清晰的边界定义和判别示例。

#### 案例 3：逻辑矛盾
```
ID: 143 | persona: 98 | 主题: 运动营养与能量补给策略
Query: 帮我看看上个月我每次练完之后，手环记录的睡眠平均心率是多少？
问题: 指定主题为"营养与能量补给"，但 query 内容是睡眠心率，主题偏移。
```

**优化方向**: 强化 Prompt 中主题锚定约束，要求 query 核心意图必须与指定主题直接相关。

### 3.4 审查结论

| 结论项 | 说明 |
|--------|------|
| 81% 正常率 | 首轮抽样即定位了 3 类主要问题，指导后续 Prompt 迭代 |
| 流程价值 | 100 条抽样 + 自动化审查 = 数分钟内完成，等效于数小时人工逐条审查 |
| 可复现性 | 同样流程可即时应用于任意 Domain，无需额外开发 |
| 优化闭环 | 问题分布直接映射到 Prompt 优化方向（术语控制 / 类型边界 / 主题锚定） |

#### 按领域的聚类质量

| 领域 | 平均准确率 | 评价 |
|------|-----------|------|

| 医疗健康 (血压/血糖) | ~60% | 语义重叠较多 |
| 中医传统 | ~85% | 特征独特，效果好 |

### 4.4 调优核心发现

1. **`min_samples=1` 是降噪的最关键参数**：从 3 降至 1，噪声率直降 50%+
2. **`n_components` 高低各有优势**：64 维细粒度但簇多，12 维粗粒度但存在大簇需细分
3. **`min_dist=0.0` 配合 `epsilon=0.08`** 是最稳定的参数组合（v1、v3 均验证）
4. **v2 方案噪声最低 (5%) 但大簇问题严重**：需二次聚类才可用

## 5. 谷歌数据集场景适配报告

### 5.1 适配概述

谷歌数据集的改写不是简单的关键词替换，而是从多个维度审视竞品生态的 query，将其适配到华为 Watch + Huawei Health 的产品场景中。

### 5.2 多维度适配分析

| 维度 | 审视角度 | 适配示例 |
|------|---------|---------|
| **品牌生态** | 识别与 Google/Fitbit 产品绑定的表述 | Fitbit → Huawei Watch |
| **AI 能力命名** | 竞品 AI 功能名称 → 华为对应功能 | Fitbit LLM → Huawei Assistant |
| **专有指标体系** | 分析竞品独有指标的定义，映射到通用/华为指标 | Active Zone Minutes → Intensity Activity Minutes（强度活动分钟数）|
| **评估模型差异** | 理解竞品健康评估模型，找到华为生态等价概念 | Readiness Score → Recovery Status（恢复状态）|
| **场景语境** | 识别隐含依赖竞品生态的场景描述 | "Fitbit 追踪睡眠数据的算法" → 华为 Watch 睡眠监测场景 |

## 7. 结论与建议

### 7.1 综合评价

| 维度 | 评价 | 等级 |
|------|------|------|
| 数据集规模 | 56,319 条多源异构，覆盖知识/运动/健康/红队多维度 | 优秀 |
| LLM 生成质量 | 去重率 0.62%，Checker 抽样 81% 正常率，问题类型可溯源 | 良好 |
| 架构可扩展性 | 新 Domain 零代码接入，生成+审查全链路自动打通 | 优秀 |
| 聚类调优 | 9 轮实验，噪声率 30%→8%，全局准确率 67.2% | 优秀 |
| 审查流程效率 | 100 条抽样即定位主要问题，等效数小时人工审查 | 优秀 |
| 谷歌场景适配 | 5 维度生态改写，语义保持，无机械替换 | 良好 |

### 7.2 优化建议

1. **Prompt 术语控制**: 根据 Checker 发现的"语言不自然"问题（47.4%），在 Prompt 中约束术语使用层次，与 persona 专业背景匹配
2. **类型边界强化**: 根据"类型不匹配"问题（31.6%），为三种 Query 类型提供更清晰的判别示例
3. **主题锚定加强**: 根据"逻辑矛盾"问题（15.8%），强化 query 核心意图与指定主题的直接关联
4. **低准确率簇优化**: 对 <30% 准确率的簇进行二次聚类或参数微调
