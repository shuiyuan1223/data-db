## 1. 测试背景与目标

### 1.1 项目概述

本项目为 HQB（Health Query Benchmark）数据集的构建系统，综合三大数据来源，构建多源异构健康查询数据集：

| 数据来源 | 原始/生成数量 | 去重/清洗后数量 | 特性描述 |
|----------|-------------|---------------|---------|
| 大模型生成 (LLM) | 49,200 | 48,895 | 结构化强，覆盖通用、显性/隐性个性化维度 |
| 谷歌数据集 (Google) | 4,170 | 7,424 | 搜索意图明确，经场景适配改写为华为生态 |
| 现网数据集 (Live) | 4,111 (抽样) | — | 真实性强，捕捉长尾与口语化表达 |
| **总计** | **57,481** | **56,319** | **多源异构** |

此外，系统内置多个可扩展的大 Domain（如 general_medical、health、sports、sports_health、red_team 等，每个约 500 条），这些 Domain 可按需持续生成、独立扩展新维度，**只需在 `prompts/` 下新增 prompt 文件并在映射表中注册，即可零代码接入生成与审查全流程**。

### 1.2 系统架构的高度可扩展性

本系统的核心设计理念是**流程标准化 + 维度可插拔**：

```
新 Domain 接入流程（零代码扩展）：
  1. prompts/ 下新建 prompt 文件 → 实现 build_messages()
  2. output_db.py 注册表 DDL → _DOMAIN_TABLE_MAP
  3. generator.py 注册 → _DOMAIN_PROMPT_MAP
  4. Checker 自动适配 → 无需额外修改
  ──────────────────────────────────────────
  新 Domain 即刻可用，生成 + 审查全链路自动打通
```

**已验证的扩展能力**：
- 现有 5 个 Domain 均通过此流程接入，每个 Domain 约 500 条且可持续追加生成
- red_team Domain 支持 4 类攻击类型，随时可扩展新攻击维度
- 知识目录 / 运动 / 健康三大板块各自独立，可按需增加类别和生成轮次

## 3. 测试策略

### 3.1 LLM 数据合成质量测试策略（三阶段保障）

全量 LLM 生成数据遵循统一规则：**1 个类别生成 30 条 Query（10 条通用 + 10 条显性个性化 + 10 条隐性个性化）**。

#### 阶段一：生成约束内嵌（Prompt Engineering）

- **三类型结构约束**: 每个类别严格生成 10 条通用 + 10 条显性个性化 + 10 条隐性个性化
- **字段合规约束**: data_fields 从预定义数据字段目录中选取
- **人设一致性约束**: query 内容与 persona 属性高度一致
- **知识主题锚定**: 每条 query 围绕指定 catalog_topic 生成

#### 阶段二：Checker 自动审查

统一质量审查系统的核心价值在于**流程自动化与人力节省**：
- 自动读取生成 prompt 约束作为审查标准，**无需为每个 Domain 人工编写审查规则**
- 分批调用 LLM reviewer（temperature=0.3），抽样审查即可快速定位质量瓶颈
- 6 维度缺陷分类：语言不自然 / 类型不匹配 / 逻辑矛盾 / 格式违规 / 重复 / 质量低下
- 输出结构化报告：按 Query 类型、知识主题、问题类型三个维度聚合统计

**架构优势**: 新增 Domain 后 Checker 零额外代码即可审查，审查标准自动与生成约束保持一致。

#### 阶段三：聚类 + LLM 准确率验证

对数据进行 UMAP+HDBSCAN 聚类后，通过 LLM 逐簇验证主题一致性，作为第二重质量保障。

### 3.2 UMAP+HDBSCAN 聚类参数调优策略

本人负责聚类参数的系统性调优，共执行 **9 轮实验**，探索 6 个参数维度：

| 版本 | n_components | min_dist | min_cluster | min_samples | epsilon | 簇数 | 噪声点 | 噪声率 |
|------|-------------|---------|------------|------------|---------|------|--------|--------|
| 初始 | 64 | 0.0 | 5 | 3 | 0.03 | 2,054 | 17,859 | ~30% |
| 方案A | 64 | 0.05 | 3 | 1 | 0.06 | 3,595 | 9,471 | ~16% |
| **v1** | **64** | **0.0** | **3** | **1** | **0.08** | **1,899** | **4,843** | **~8%** |
| v2 | 12 | 0.12 | 3 | 1 | 0.18 | 883 | 3,080 | ~5% |
| **v3** | **12** | **0.0** | **3** | **1** | **0.08** | **1,823** | **4,761** | **~8%** |

**参数影响规律**:

| 参数 | 降低 → 效果 | 增加 → 效果 |
|------|------------|------------|
| epsilon | 更多噪声 | 更少噪声 |
| min_samples | 更少噪声 | 更多噪声 |
| min_cluster_size | 更少噪声 | 更多噪声 |
| min_dist | 点聚集，边界模糊 | 点分散，识别更清晰 |

**调优结论**: 噪声率从初始 ~30% 降至最终 ~8%（降幅 73%），v1/v3 为最终推荐方案。

### 3.3 谷歌数据集场景适配测试策略

谷歌数据集的改写**不是简单的关键词替换，而是多维度的生态场景适配**：

**适配维度与审视角度**：

| 维度 | 审视角度 | 举例 |
|------|---------|------|
| 品牌生态 | 识别 Google/Fitbit 生态的产品绑定 | Fitbit → Huawei Watch |
| AI 能力命名 | 区分竞品 AI 功能名称 vs 华为对应功能 | Fitbit LLM → Huawei Assistant |
| 专有指标体系 | 分析竞品独有的健康指标定义，映射到华为/通用指标 | Active Zone Minutes (AZM) → Intensity Activity Minutes (强度活动分钟数) |
| 评估模型差异 | 理解竞品特有的健康评估模型，找到华为生态的等价概念 | Readiness Score → Recovery Status (恢复状态) |
| 场景语境适配 | 识别隐含依赖 Google 生态的场景描述 | "Fitbit 用于追踪睡眠数据的具体算法" → 华为 Watch 睡眠监测场景 |

**改写原则**：
1. **语义优先**: 不是机械替换文本，而是理解 query 背后的用户意图，找到华为生态中语义等价的表述
2. **指标对齐**: 竞品专有指标需分析其定义（如 AZM 的心率区间逻辑），再选择华为/通用中最接近的概念
3. **场景还原**: 改写后的 query 必须在华为 Watch + Huawei Health 场景下逻辑自洽
4. **LLM 辅助筛选**: 通过 LLM 识别肉眼难以发现的隐含竞品关联（非直接命名的 Google 相关场景）

## 7. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 输出不稳定 | 质量波动 | Checker 自动审查 + 温度控制(0.3) |
| API 限流/超时 | 大批量生成中断 | 指数退避重试 + checkpoint 断点续传 |
| 数据字段幻觉 | 虚假字段 | Prompt 硬约束 + Checker 格式违规检测 |
| 聚类大簇 | 粒度不均 | 参数调优（v1/v3）+ 二次聚类 |
| 改写遗漏 | 竞品术语残留 | 多关键词扫描 + LLM 辅助识别 |
