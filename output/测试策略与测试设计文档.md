## 1. 测试背景与目标

### 1.1 项目概述

本人围绕已有的健康查询基准数据库，构建了一套**端到端的 AI 数据自动化生态系统**，覆盖从 Prompt 驱动的数据合成、多源数据清洗与场景适配、UMAP+HDBSCAN 聚类调优，到统一质量审查的全链路。

本系统的核心价值不在于单个模块，而在于**生成 → 聚类 → 审查的一条龙自动化数据管理能力**：

```
┌────────────────────────────────────────────────────────────────┐
│                 端到端数据自动化生态系统                          │
│                                                                │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐  │
│  │ Prompt   │ →  │ LLM 数据 │ →  │ 聚类     │ →  │ 统一    │  │
│  │ 工程     │    │ 自动合成 │    │ 参数调优 │    │ 质量审查│  │
│  │ (5 Domain│    │ (async+  │    │ (UMAP+   │    │ (Checker│  │
│  │  可扩展) │    │ checkpoint)   │ HDBSCAN) │    │  自适应)│  │
│  └──────────┘    └──────────┘    └──────────┘    └─────────┘  │
│       ↑                                               │        │
│       └───────── 审查结果反馈 Prompt 优化 ────────────┘        │
└────────────────────────────────────────────────────────────────┘
```

### 1.3 Domain 可插拔架构

本系统设计了 **Domain 可插拔架构**，每个 Domain 都是独立的数据生成维度，拥有独立的 Prompt 模板，可持续生成且随时扩展新维度：

| Domain | 独立 Prompt 文件 | 估计规模 | 可扩展性 |
|--------|----------------|---------|---------|
| general_medical | `prompts/general_medical.py` | ~500 条 | 可持续追加生成轮次，扩展新知识主题 |
| health | `prompts/health.py` | ~500 条 | 可持续追加，支持 topic_count 多子域 |
| sports | `prompts/sports.py` | ~500 条 | LLM 内联 persona，可扩展运动项目 |
| sports_health | `prompts/sports_health.py` | ~500 条 | 跨域 persona 适配，可扩展交叉场景 |
| red_team | `prompts/red_team.py` | ~500 条 | 4 类攻击维度，可扩展新攻击类型 |

**新 Domain 接入流程（零代码扩展框架）**：
```
1. prompts/ 新建 prompt 文件 → 实现 build_messages()
2. output_db.py 注册表 DDL → _DOMAIN_TABLE_MAP
3. generator.py 注册 → _DOMAIN_PROMPT_MAP
4. Checker 自动适配 → 无需额外代码
─────────────────────────────────
新 Domain 即刻可用，生成 + 审查全链路自动打通
```

## 3. 测试策略

### 3.1 生态系统端到端测试策略

本系统的核心测试策略是验证**生成→聚类→审查**的一条龙闭环：

```
Prompt 设计 → LLM 批量生成 → 聚类分析数据分布
     ↑                              ↓
     └──── Checker 审查发现问题 ←── 聚类验证质量
           → 反馈 Prompt 优化
```

### 3.2 LLM 数据合成质量测试策略

全量生成规则：**1 个类别 30 条 Query（10 通用 + 10 显性个性化 + 10 隐性个性化）**

#### 阶段一：Prompt 约束内嵌

每个 Domain 的独立 Prompt 文件中内嵌质量约束：
- **三类型结构约束**: 每类别严格 10+10+10
- **字段合规约束**: data_fields 从预定义目录选取
- **人设一致性约束**: query 与 persona 属性匹配
- **知识主题锚定**: query 围绕指定 topic 生成

#### 阶段二：Checker 自动审查（核心流程价值）

Checker 的设计目标是**以最小人力投入快速定位质量瓶颈**：
- 自动读取任意 Domain 的 Prompt 约束作为审查标准
- 抽样 100 条即可定位主要问题类型（等效数小时人工审查）
- 6 维度缺陷分类 + 三维度聚合报告 → 直接指导 Prompt 优化
- **新 Domain 零额外代码即可审查**

#### 阶段三：聚类验证

对数据进行聚类后通过 LLM 逐簇验证主题一致性，作为第二重质量保障。

### 3.3 聚类参数调优策略

本人负责聚类参数的系统性调优，共执行 **9 轮实验**，探索 6 个参数维度：

| 版本 | n_comp | min_dist | min_clust | min_samp | eps | 簇数 | 噪声率 |
|------|--------|---------|-----------|----------|-----|------|--------|
| 初始 | 64 | 0.0 | 5 | 3 | 0.03 | 2,054 | ~30% |
| 方案A | 64 | 0.05 | 3 | 1 | 0.06 | 3,595 | ~16% |
| **v1** | **64** | **0.0** | **3** | **1** | **0.08** | **1,899** | **~8%** |
| v2 | 12 | 0.12 | 3 | 1 | 0.18 | 883 | ~5% |
| **v3** | **12** | **0.0** | **3** | **1** | **0.08** | **1,823** | **~8%** |

**参数影响规律**:

| 参数 | 降低 → | 增加 → |
|------|--------|--------|
| epsilon | 更多噪声 | 更少噪声 |
| min_samples | 更少噪声 | 更多噪声 |
| min_cluster_size | 更少噪声 | 更多噪声 |
| min_dist | 点聚集，边界模糊 | 点分散，识别清晰 |

### 3.4 谷歌数据多维度场景适配策略

改写**不是机械的关键词替换**，而是从 5 个维度审视竞品生态语境后的语义等价适配：

| 维度 | 审视角度 |
|------|---------|
| 品牌生态绑定 | 识别与 Google/Fitbit 产品绑定的表述 |
| AI 功能命名 | 竞品 AI 功能 vs 华为对应功能的差异 |
| 专有指标体系 | 分析竞品独有指标的定义与计算逻辑，映射通用/华为指标 |
| 健康评估模型 | 理解竞品评估模型含义，找到华为生态等价概念 |
| 隐含场景语境 | LLM 辅助识别非直接命名的竞品关联场景 |

### 3.5 系统可靠性测试策略

| 测试场景 | 预期结果 |
|----------|---------|
| --batch 10 并发生成 | 无死锁、无数据竞争 |
| 中途 kill 后重启 | checkpoint 跳过已完成任务 |
| API 限流 | 指数退避（2s→30s cap）自动恢复 |

## 6. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| LLM 输出不稳定 | Checker 自动审查 + 温度控制(0.3) |
| API 限流/超时 | 指数退避重试 + checkpoint 断点续传 |
| 数据字段幻觉 | Prompt 硬约束 + Checker 格式违规检测 |
| 聚类大簇 | 参数调优（v1/v3）+ 二次聚类 |
| 新 Domain 质量不稳 | Checker 自适应审查，首轮即定位问题 |
