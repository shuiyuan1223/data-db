## 1. 测试背景与目标

### 1.1 项目概述

本人围绕已有的健康查询基准数据库，构建了一套**端到端的 AI 数据自动化生态系统**，覆盖从 Prompt 驱动的数据合成、多源数据清洗与场景适配、UMAP+HDBSCAN 聚类调优，到统一质量审查的全链路。

本系统的核心价值不在于单个模块，而在于**生成 → 聚类 → 审查的一条龙自动化数据管理能力**：

```
┌────────────────────────────────────────────────────────────────┐
│                 端到端数据自动化生态系统                          │
│                                                                │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐  │
│  │ Prompt   │ →  │ LLM 数据 │ →  │ 聚类     │ →  │ 统一    │  │
│  │ 工程     │    │ 自动合成 │    │ 参数调优 │    │ 质量审查│  │
│  │ (5 Domain│    │ (async+  │    │ (UMAP+   │    │ (Checker│  │
│  │  可扩展) │    │ checkpoint)   │ HDBSCAN) │    │  自适应)│  │
│  └──────────┘    └──────────┘    └──────────┘    └─────────┘  │
│       ↑                                               │        │
│       └───────── 审查结果反馈 Prompt 优化 ────────────┘        │
└────────────────────────────────────────────────────────────────┘
```

每个 Domain 都是独立的数据生成维度，拥有独立的 Prompt 模板文件（`prompts/*.py`），可持续生成且随时扩展新维度：

**新 Domain 接入流程（零代码扩展框架）**：
```
1. prompts/ 新建 prompt 文件 → 实现 build_persona_block() + build_messages()

4. Checker 自动适配 → 无需额外代码
─────────────────────────────────
新 Domain 即刻可用，生成 + 审查全链路自动打通
```

## 3. 测试策略

### 3.1 生态系统端到端测试策略

核心测试策略是验证**生成→聚类→审查**的一条龙闭环：

```
Prompt 设计 → LLM 批量生成 → 聚类分析数据分布
     ↑                              ↓
     └──── Checker 审查发现问题 ←── 聚类验证质量
           → 反馈 Prompt 优化
```

### 3.2 LLM 数据合成质量测试策略

全量生成规则：**1 个类别 30 条 Query（10 通用 + 10 显性个性化 + 10 隐性个性化）**

#### 阶段一：Prompt 约束内嵌

每个 Domain 的 Prompt 文件中，通过 `build_messages()` 函数构造 system + user 消息对，在 SYSTEM_PROMPT 中内嵌以下约束：

- **三类型结构约束**: Type 1 = 通用知识查询（不涉及个人数据）；Type 2 = 显性个性化（直接请求 1-3 个数据字段）；Type 3 = 隐性个性化（隐含需要 2-4 个数据字段做分析计算）
- **字段合规约束**: 在 general_medical 中维护 101 个字段的 DATA_FIELD_CATALOG（基础信息/体成分/每日汇总/心率压力/睡眠/健康档案/营养/高频时序 8 大类），health 维护 52 个字段
- **人设一致性约束**: `build_persona_block()` 将 persona 的年龄/性别/BMI/健康关注/运动偏好等结构化渲染为 LLM 可读的卡片
- **术语适配约束**: Prompt 要求 LLM 根据 persona 的医学知识水平调整术语——普通用户用日常语言，医疗专业人员可用专业术语

#### 阶段二：Checker 自动审查

Checker（`checker.py`）是本生态系统的**质量闭环引擎**：

1. `_extract_domain_info(domain)` 自动从对应 Domain 的 prompt 模块中提取 SYSTEM_PROMPT 和 query_type 定义
2. `_build_review_prompt()` 将提取的生成约束注入审查 prompt，让 LLM reviewer 理解原始质量标准
3. 分批审查（默认 batch_size=30），temperature=0.3 低方差模式
4. 6 维度缺陷分类：语言不自然 / 类型不匹配 / 逻辑矛盾 / 格式违规 / 重复 / 质量低下
5. `_write_report()` 输出按 Query 类型 / 知识主题 / 问题类型三维聚合的结构化报告

**关键架构优势**: 新增 Domain 只要在 `_DOMAIN_PROMPT_MAP` 中注册了 prompt 模块，Checker 就能自动审查，无需额外代码。

#### 阶段三：聚类验证

对数据进行 UMAP+HDBSCAN 聚类后，通过 LLM 逐簇验证主题一致性，作为第二重质量保障。

### 3.3 聚类参数调优策略

本人负责聚类参数的系统性调优，共执行 **9 轮实验**，探索 UMAP（n_neighbors / n_components / min_dist）和 HDBSCAN（min_cluster_size / min_samples / epsilon / method）共 6+ 个参数维度：

| 版本 | n_comp | min_dist | min_clust | min_samp | eps | 簇数 | 噪声率 |
|------|--------|---------|-----------|----------|-----|------|--------|
| 初始 | 64 | 0.0 | 5 | 3 | 0.03 | 2,054 | ~30% |
| 方案A | 64 | 0.05 | 3 | 1 | 0.06 | 3,595 | ~16% |
| **v1** | **64** | **0.0** | **3** | **1** | **0.08** | **1,899** | **~8%** |
| v2 | 12 | 0.12 | 3 | 1 | 0.18 | 883 | ~5% |
| **v3** | **12** | **0.0** | **3** | **1** | **0.08** | **1,823** | **~8%** |

**参数影响规律**:

| 参数 | 降低 → | 增加 → |
|------|--------|--------|
| epsilon | 更多噪声（HDBSCAN 对密度要求更严格） | 更少噪声（允许更松散的点归入簇） |
| min_samples | 更少噪声（核心点判定更宽松） | 更多噪声（核心点判定更严格） |
| min_cluster_size | 更少噪声（允许更小的簇） | 更多噪声（小簇被判定为噪声） |
| min_dist (UMAP) | 嵌入点更聚集，簇边界模糊 | 嵌入点更分散，簇边界更清晰 |
| n_components (UMAP) | 更少簇（粗粒度，但大簇多） | 更多簇（细粒度，信息保留多） |

### 3.4 谷歌数据多维度场景适配策略

改写**不是机械的关键词替换**，而是从 5 个维度审视竞品生态语境后的语义等价适配：

| 维度 | 审视角度 | 适配示例 |
|------|---------|---------|
| 品牌生态绑定 | 识别与 Google/Fitbit 产品绑定的表述 | Fitbit → Huawei Watch |
| AI 功能命名 | 竞品 AI 功能 vs 华为对应功能的差异 | Fitbit LLM → Huawei Assistant |
| 专有指标体系 | 分析竞品独有指标的定义与计算逻辑 | Active Zone Minutes (基于心率区间) → Intensity Activity Minutes |
| 健康评估模型 | 理解竞品评估模型含义，找等价概念 | Readiness Score (综合HRV/睡眠/活动) → Recovery Status |
| 隐含场景语境 | LLM 辅助识别非直接命名的竞品关联 | "Fitbit 追踪睡眠的算法" → 华为 Watch 睡眠监测场景 |

### 3.5 系统可靠性测试策略

系统可靠性基于 `llm_client.py` 的指数退避和 `generator.py` 的 checkpoint 机制：

| 测试场景 | 实现机制 | 预期结果 |
|----------|---------|---------|
| 并发稳定性（--batch 10） | asyncio.Semaphore 限流 + asyncio.Lock 保护 DB 写入 | 无死锁、无数据竞争 |
| 断点续传 | `is_generated()`/`is_topic_generated()` 检查 persona×topic 是否已有数据 | 跳过已完成任务 |
| 指数退避重试 | `delay = min(2 × 2^(retry-1), 30)` 秒，默认最大重试 100 次 | API 限流下自动恢复 |
| 失败记录 | `_record_failure()` → failed_tasks 表 (domain/persona/topic/error/attempts/timestamp) | 异常完整可溯 |

## 6. 风险与缓解

| 风险 | 实现层缓解机制 | 测试层验证 |
|------|-------------|-----------|
| LLM 输出不稳定 | Checker 自动审查(temp=0.3) + Prompt 约束 | 抽样审查正常率 ≥ 80% |
| API 限流/超时 | `delay = min(2×2^(n-1), 30)` 指数退避 + checkpoint | 断点续传测试 |
| 数据字段幻觉 | DATA_FIELD_CATALOG 硬约束(101/52字段) | Checker 格式违规检测 |
| 聚类大簇 | 参数调优(v1/v3) + 二次聚类 | 噪声率 < 10% |
| 新 Domain 质量不稳 | Checker `_extract_domain_info` 自适应 | 首轮抽样即定位问题 |
| DB 写入竞争 | asyncio.Lock 保护 + SQLite WAL | 并发写入无重复 |
